## 一、RMT — 需求管理团队

### 定义与定位
**RMT（Requirement Management Team）** 是需求管理的**驱动者和决策责任团队**，拥有需求接纳/拒绝/分发的最终决策权。

### 核心职责（五项）
1. 📋 **需求决策**：审视 RAT 分析结果，决策是否接纳、优先级排序、何时交付
2. 🗂️ **需求分发管理**：监控需求从 IR 到版本/路标的分发执行情况
3. 🔍 **重要需求跟踪**：跟踪已承诺重要需求的实现进展及风险
4. 🔗 **跨产品线协调**：协调本领域内跨产品线需求的资源和优先级
5. 📊 **需求质量管理**：监控需求管理过程质量，推动持续改进

### 决策依据
需求对**客户的价值**和对**产品竞争力提升**的价值，而非个人影响力或内部政治。

### 团队成员构成（跨部门）

| 角色 | 职责 |
|------|------|
| **RMT 主任**（产品管理部负责人）| 会议主持，最终决策拍板 |
| 产品管理代表 | 市场视角，需求价值判断 |
| 开发代表 | 技术可行性评估 |
| 系统工程师（SE）代表 | 需求规格把关 |
| 市场/营销代表 | 客户反馈和竞品信息 |
| 技术服务代表 | 服务/维护层面的需求 |
| 质量与运营代表 | 质量要求和运营约束 |

### 层级分类

| 级别 | 名称 | 职责范围 |
|------|------|---------|
| 🏢 **C-RMT** | 公司级 RMT | 管理公司级重大项目及跨产品线需求；协调公司级资源 |
| 📦 **PL-RMT** | 产品线级 RMT | 负责产品线日常需求管理和分发监控；一般每月召开例会 |

### 例会机制
- **频率**：PL-RMT 每月至少一次，重大版本节点前加开专题会
- **输入**：RAT 分析结论 + 需求价值评估报告
- **输出**：需求接纳决策记录 + 版本分发计划

## 二、RAT — 需求审核团队

### 定义与定位
**RAT（Requirement Analysis Team）** 是 RMT 的**专业分析支撑团队**，负责对原始需求进行系统化分析，给出接纳建议，但**不做最终决策**（决策权在 RMT）。

### 核心职责（五项）
1. 🔬 **批量专业分析**：对 RR 原始需求进行解释/过滤/分类/排序/证实
2. 📑 **市场调研**：必要时开展市场调研，验证需求真实性和价值
3. 📋 **评估报告输出**：给出收益程度/风险程度/工作量/是否采纳的建议
4. 📦 **市场需求包整理**：形成结构化需求包，提交 RMT 决策
5. 🔁 **早期确认跟踪**：跟踪客户需求早期确认工作进展

### 核心分析活动（五步）

```
原始需求（RR）
      ↓
1️⃣ 解释 — 澄清还原真实业务场景
      ↓
2️⃣ 过滤 — 剔除无价值/不可行的需求
      ↓
3️⃣ 分类 — 按领域/类型/紧急程度归类
      ↓
4️⃣ 排序 — 基于价值/工作量/战略符合度评估
      ↓
5️⃣ 证实 — 必要时开展市场调研验证
      ↓
输出：初始需求清单（IR）+ 需求价值评估报告
```

### 团队成员构成

| 角色 | 职责 |
|------|------|
| 产品管理代表 | 市场分析牵头 |
| **需求管理工程师（REM）** | 分析流程组织，记录管理 |
| 开发代表 | 技术可行性预判 |
| 系统工程师（SE）| 需求规范化指导 |
| 市场代表 | 客户背景信息 |
| 技术服务代表 | 服务/维护视角 |

### 层级
- **PL-RAT**（产品线级）：在 PL-RMT 组织下，按领域/子产品线/产品族设立多个并行 RAT

## 三、RMT 与 RAT 协同关系

```
客户/市场/内部原始需求（RR）
              ↓
         ┌─────────────────────────────┐
         │  【RAT】专业分析             │
         │  · 澄清真实业务场景          │
         │  · 评估价值/风险/工作量      │
         │  · 给出采纳建议和排序建议    │
         └─────────────────────────────┘
              ↓ 输出：IR + 价值评估报告
         ┌─────────────────────────────┐
         │  【RMT】决策                 │
         │  · 审视 RAT 分析结论         │
         │  · 决策是否接纳              │
         │  · 确定优先级和版本分发计划  │
         └─────────────────────────────┘
              ↓
         分发到：PDT / TDT / 产品路标 / SP-BP 规划
```

## 四、在诊断中评估 RMT/RAT 成熟度

| 评估维度 | L1 表现 | L3 目标状态 |
|----------|---------|------------|
| 组织建立 | 无 RMT/RAT，需求由研发或销售个人决定 | RMT/RAT 有正式章程，成员明确 |
| 例会机制 | 无例会，需求随时插入 | 每月例会，有明确输入输出 |
| 决策记录 | 无记录，靠记忆或口头承诺 | 需求决策有正式记录，可追溯 |
| 分析质量 | 原始需求直接变成开发任务 | 需求经过 RAT 五步分析，有价值评估 |
| 跨部门参与 | 仅研发视角，市场/销售不参与 | 跨职能团队，市场/技术/服务均有代表 |
