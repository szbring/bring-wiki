# DCP决策评审机制

## 一句话定义
DCP（Decision Check Point）是[[IPD]]流程中的商业投资决策关口，在每个阶段结束时由IPMT做出Go/Kill/Hold决策，确保资源只投入在值得的项目上。

---

<!-- chunk: dcp.core-logic -->
## DCP的核心逻辑

```
概念阶段 ──→ CDCP ──→ 计划阶段 ──→ PDCP ──→ 开发阶段 ──→ ADCP ──→ 验证阶段 ──→ LDCP ──→ 发布阶段 ──→ EDCP
              │                    │                    │                    │                    │
           值得做吗？           计划可行吗？          做对了吗？           质量达标吗？         继续投资吗？
           (投入5%)            (投入15%)            (投入50%)           (投入85%)           (持续运营)
```

**资源投入规律**：每个DCP之前只投入少量资源验证假设，通过后才追加投资。越往后发现问题的成本越高——DCP的核心价值在于"尽早发现方向性错误"。

---

## 五个DCP详解

<!-- chunk: dcp.cdcp -->
### CDCP — 概念评审（立项评审）

| 维度 | 内容 |
|---|---|
| 评审时机 | 概念阶段结束时 |
| 核心问题 | 这个产品值得做吗？市场机会真实存在吗？ |
| 评审重点 | 市场机会分析、初步产品定义、商业可行性 |
| 通过标准 | 市场机会真实、产品方向正确、商业回报初步可行 |
| 此时资源投入 | 约占总投入的5% |

**CDCP必须回答的5个问题**：
1. 目标客户是谁？痛点是否真实？
2. 市场规模是否足够大？
3. 竞争格局如何？我们能赢吗？
4. 技术上是否可行？
5. 初步财务回报是否合理？

<!-- chunk: dcp.pdcp -->
### PDCP — 计划评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 计划阶段结束时 |
| 核心问题 | 方案和计划可行吗？资源到位了吗？ |
| 评审重点 | 详细技术方案、完整项目计划、资源落实确认 |
| 通过标准 | 方案可行、计划完整、资源到位、风险可控 |
| 此时资源投入 | 约占总投入的15% |

**PDCP的关键检查项**：
- 技术方案评审通过（TR2）
- 项目计划完整且经过各部门确认
- 核心人员已到位
- 关键物料/供应商已确认
- 预算已审批

<!-- chunk: dcp.adcp -->
### ADCP — 开发评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 开发阶段结束时 |
| 核心问题 | 开发目标达成了吗？可以进入测试了吗？ |
| 评审重点 | 开发完成度、代码/设计质量、测试就绪度 |
| 通过标准 | 开发完成、单元测试通过、测试环境就绪 |
| 此时资源投入 | 约占总投入的50% |

<!-- chunk: dcp.ldcp -->
### LDCP — 发布评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 验证阶段结束时 |
| 核心问题 | 产品质量达标了吗？可以上市了吗？ |
| 评审重点 | 测试结果、质量问题关闭率、上市准备就绪度 |
| 通过标准 | 全部P0/P1缺陷关闭、P2缺陷有规避方案、上市物料就绪 |
| 此时资源投入 | 约占总投入的85% |

<!-- chunk: dcp.edcp -->
### EDCP — 生命周期评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 产品上市后，每年至少一次 |
| 核心问题 | 产品是否继续投资？何时退市？ |
| 评审重点 | 产品销售数据、市场份额、盈利状况、竞争态势 |
| 决策选项 | 继续投资 / 优化调整 / 计划退市 |

<!-- chunk: dcp.decision-types -->
## DCP决策机制

### 三种决策结果

| 决策 | 含义 | 后续动作 |
|---|---|---|
| **Go** | 继续 | 进入下一阶段，追加资源投入 |
| **Kill** | 终止 | 项目终止，释放资源，做善后处理 |
| **Hold** | 暂停 | 暂停项目，等待条件变化后重新评审 |

### 不接受的"第四种决策"
- ~~"有条件通过"~~ → 必须明确条件是什么，满足后重新评审
- ~~"先做着看"~~ → 这不是决策，是逃避决策
- ~~"小规模试一下"~~ → 必须明确试什么、到什么标准算通过

### Kill决策的勇气
华为任正非强调：**"Kill一个项目，比Go一个项目更有价值"**。Kill节省的是后续80%的沉没成本。华为历史上Kill掉的项目远多于Go的项目。

<!-- chunk: dcp.review-process -->
## DCP评审流程

```
PDT提交DCP材料（提前3-5个工作日）
        │
        ↓
IPMT成员预审（独立审阅，形成初步意见）
        │
        ↓
DCP评审会议（2-4小时）
  ├─ PDT汇报（30-60分钟）
  ├─ IPMT提问与讨论（60-120分钟）
  └─ IPMT闭门决策（15-30分钟）
        │
        ↓
决策通知（24小时内书面发出）
  ├─ Go → 进入下一阶段
  ├─ Kill → 终止项目，资源释放
  └─ Hold → 暂停，明确恢复条件
```

<!-- chunk: dcp.vs-tr -->
## DCP与TR的关系

```
DCP（商业决策）      TR（技术评审）
    │                    │
 由IPMT决策          由IRT评审
    │                    │
 关注"值不值得做"    关注"做得对不对"
    │                    │
 Go/Kill/Hold         通过/不通过
```

**关键规则**：TR不通过，不能进入DCP评审。技术可行性是商业决策的前提。

<!-- chunk: dcp.pitfalls -->
## 常见问题与应对

| 问题 | 表现 | 应对 |
|---|---|---|
| DCP走过场 | IPMT不提问就通过 | 设定必问问题清单，顾问参与前3次评审 |
| Kill不下去 | 明知项目有问题但不舍得终止 | 用数据说话：已投入是沉没成本，继续投入是追加损失 |
| DCP材料不齐 | PDT赶进度，材料敷衍 | 明确材料提交时限，不提交不排评审 |
| 频繁Hold | 项目反复暂停又恢复 | 设定Hold最长期限，超期自动Kill |
| IPMT成员缺席 | 评审会人到不齐 | 设定法定人数（≥2/3），不到不排会 |

## 薄云咨询的DCP落地建议

### 中小企业简化方案
不必照搬华为的5个DCP，可以简化为3个关键评审点：

| 简化方案 | 对应华为DCP | 评审重点 |
|---|---|---|
| DCP-立项 | CDCP+PDCP | 值得做+计划可行 |
| DCP-开发完成 | ADCP+LDCP | 做完了+质量达标 |
| DCP-年度评审 | EDCP | 继续投资还是退市 |

### 落地优先级
1. 先把CDCP（立项评审）做实——这是ROI最高的评审点
2. 再做LDCP（发布评审）——防止质量不合格的产品上市
3. 最后补齐其他DCP
