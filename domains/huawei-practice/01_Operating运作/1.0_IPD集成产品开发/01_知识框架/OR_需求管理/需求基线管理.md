# 需求基线管理

> 所属流程：[[OR需求管理流程概述]]
> 标签：#华为 #[[IPD]] #基线管理 #变更控制 #CCB

---

<!-- chunk: requirement-baseline.一句话定义 -->
## 一句话定义
需求基线是经正式评审和决策后**被冻结**的需求版本，是后续开发、测试、验收的唯一依据。基线管理的本质是**门禁式管控**：未经评审不得随意变更。

---

<!-- chunk: requirement-baseline.一-基线建立条件 -->
## 一、基线建立条件

| 条件 | 说明 |
|------|------|
| ✅ 完成需求分析 | RAT 完成解释/过滤/分类/排序，需求已标准化（IR/SF/SR 层）|
| ✅ 完成 RMT 决策 | RMT 例会已评审，确认需求接纳意见和版本分发计划 |
| ✅ Charter/CDBT 评审通过 | 需求包已纳入产品任务书，且通过评审 |
| ✅ TR1 技术评审通过 | 概念阶段需求和业务方案评审通过 |
| ✅ IPMT/产品线管理层批准 | 商业决策层面确认投资方向 |

**建立时机**：
- **概念基线**：CDCP 通过后建立（冻结 IR/SF 层）
- **计划基线**：PDCP 通过后建立（冻结 SR 层）
- **发布基线**：ADCP 通过后建立（冻结产品规格书）

---

<!-- chunk: requirement-baseline.二-冻结规则 -->
## 二、冻结规则

| 规则 | 说明 |
|------|------|
| **TR1 前冻结概念阶段基线** | CDCP 通过后，OR/SF 层需求冻结，变更需走变更流程 |
| **TR2/TR3 前冻结计划阶段基线** | PDCP 通过后，SR 层需求规格冻结 |
| **Must 级别需求全程冻结** | MoSCoW-M 级需求纳入版本后，**不得随意删减** |
| **已对客户承诺的需求冻结** | 已写入合同/SLA/产品规格书的需求，变更需客户书面确认 |
| **变更窗口期** | 每个版本计划冻结点（FP）前接受变更，之后原则上不接受 |

---

<!-- chunk: requirement-baseline.三-变更触发机制 -->
## 三、变更触发机制

| 触发条件 | 变更级别 | 处理方式 |
|----------|----------|---------|
| 客户提出新需求/变更现有需求 | 🟢 低 | 评估后纳入需求池/后续版本 |
| 市场竞争/技术演进需要调整 | 🟡 中 | 评估对进度/成本影响，RMT 决策 |
| 重大设计缺陷/技术方案颠覆 | 🔴 高 | 触发 **ECN（工程变更通知）**，需 CCB 评审 |
| 已冻结基线的 Must/O 级需求变更 | 🚨 重大 | 触发 **BCR（基线变更请求）**，必须 CCB 评审 |
| Charter 方向/范围重大变化 | 🚨🚨 紧急 | 需回到 Charter 评审，IPMT 重新决策 |

<!-- chunk: requirement-baseline.四-ccb-变更控制委员会运作 -->
## 四、CCB 变更控制委员会运作

### CCB 构成
- **主席**：产品线总监/PDT Leader（视变更影响范围确定）
- **常设成员**：需求经理、系统工程师、开发代表、测试代表、产品经理
- **视需参与**：市场代表、财务代表、质量代表

### 变更评审流程

```
变更请求提交（含：变更原因/影响范围/紧迫程度）
         ↓
CCB 秘书登记，分配变更编号（CR-XXXX）
         ↓
初步影响分析（范围/进度/成本/质量/风险）
         ↓
CCB 会议评审
  · 评估变更必要性（是否真的需要变？）
  · 评估变更影响（版本范围/进度/资源/已测试内容）
  · 确定变更方案（接受/拒绝/延迟/分批实施）
  · 决策变更优先级
         ↓
CCB 决策输出：
  ✅ 批准 → 更新基线 → 通知相关团队 → 跟踪实施
  ❌ 拒绝 → 维持原基线 → 告知变更请求方原因
  ⏸️ 推迟 → 纳入下一版本规划
```

### CCB 评审关注点
1. **必要性**：这个变更是否真的必须在本版本做？能推迟吗？
2. **影响评估**：会影响哪些已完成的需求/设计/测试？
3. **资源可行性**：是否有足够时间和人力实现？
4. **风险**：变更引入的新风险是否可控？

<!-- chunk: requirement-baseline.五-基线管理最佳实践 -->
## 五、基线管理最佳实践

| 实践 | 说明 |
|------|------|
| **"一刀切"原则** | 基线一旦冻结，任何变更必须走正式流程，不允许"小改不走流程" |
| **变更影响可视化** | 建立需求追踪矩阵（RTM），变更时自动识别关联影响范围 |
| **版本树管理** | 每次基线变更产生新版本号，历史版本保留，可随时回溯 |
| **变更成本意识** | 统计每次变更的成本（工时/延期/返工），让团队理解随意变更的代价 |
| **变更窗口期制度** | 明确规定"特性冻结日期"，冻结后仅接受 P1 级以上变更 |
