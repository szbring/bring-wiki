# TR技术评审体系

<!-- chunk: tr-mechanism.一句话定义 -->
## 一句话定义
TR（Technical Review）是[[IPD]]流程中的技术质量关卡，在每个阶段的关键技术节点由IRT评审，确保技术方案正确、质量达标，为DCP商业决策提供技术可行性支撑。

---

<!-- chunk: tr-mechanism.tr与dcp的关系 -->
## TR与DCP的关系

```
概念阶段 ──TR1──→ 计划阶段 ──TR2──→TR3──→ 开发阶段 ──TR4──→TR5──→ 验证阶段 ──TR6──→ 发布阶段
             │                │          │             │          │             │
          系统方案         详细方案    规格     开发完成    样机验证    发布就绪
          评审             评审       确定     评审        评审       评审
             │                │          │             │          │             │
           DCP0             DCP1                    DCP2        DCP3         DCP4
```

**核心规则**：TR不通过，不能进入对应的DCP评审。技术评审是商业决策的"前置门禁"。

<!-- chunk: tr-mechanism.六个tr详解 -->
## 六个TR详解

### TR1 — 概念评审（系统方案评审）

| 维度 | 内容 |
|---|---|
| 评审时机 | 概念阶段中后期 |
| 核心问题 | 系统方案方向对不对？技术上是否可行？ |
| 评审重点 | 系统架构方案、关键技术可行性、技术风险识别 |
| 输入 | 产品需求包、初始系统架构、关键技术分析报告 |
| 输出 | 系统架构基线、关键技术决策、风险清单 |

**TR1关键检查项**：
- [ ] 系统架构是否满足产品需求
- [ ] 关键技术路径是否验证
- [ ] 技术风险是否识别并有应对方案
- [ ] 是否有未解决的技术不确定性

### TR2 — 计划评审（详细方案评审）

| 维度 | 内容 |
|---|---|
| 评审时机 | 计划阶段早期 |
| 核心问题 | 详细技术方案是否完整？各子系统接口是否定义清楚？ |
| 评审重点 | 详细设计、子系统分解、接口定义、测试策略 |
| 输入 | 系统架构基线、详细设计文档、子系统规格 |
| 输出 | 详细设计基线、子系统接口协议、测试计划 |

### TR3 — 规格确定评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 计划阶段后期，DCP1之前 |
| 核心问题 | 产品规格是否冻结？所有需求是否已分配到子系统？ |
| 评审重点 | 产品规格基线、需求追踪矩阵、子系统规格对齐 |
| 输入 | 详细设计、需求追踪矩阵、各子系统规格 |
| 输出 | 产品规格基线、需求分配表、子系统规格基线 |

**TR3是计划阶段的"技术关门点"**——规格冻结后，不再接受需求变更（变更走正式变更流程）。

### TR4 — 开发完成评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 开发阶段后期 |
| 核心问题 | 开发工作是否完成？代码/设计质量如何？ |
| 评审重点 | 代码完成度、单元测试覆盖率、集成测试就绪度 |
| 输入 | 代码/设计文档、单元测试报告、代码审查记录 |
| 输出 | 开发完成确认、集成测试准入判定 |

**TR4关键指标**：
- 代码完成度：100%（所有计划功能已编码）
- 单元测试覆盖率：≥80%
- 代码审查：关键模块100%经过评审
- 静态分析：无高优先级缺陷

### TR5 — 样机验证评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 验证阶段中期 |
| 核心问题 | 样机是否满足规格要求？测试结果是否合格？ |
| 评审重点 | 集成测试结果、系统测试结果、缺陷分析和修复情况 |
| 输入 | 测试报告、缺陷清单、回归测试结果 |
| 输出 | 验证结果确认、遗留问题清单、发布风险评估 |

### TR6 — 发布就绪评审

| 维度 | 内容 |
|---|---|
| 评审时机 | 验证阶段后期，DCP3之前 |
| 核心问题 | 产品质量是否达到发布标准？ |
| 评审重点 | 全部测试完成度、缺陷关闭率、量产/交付就绪度 |
| 输入 | 完整测试报告、缺陷关闭报告、量产准备报告 |
| 输出 | 发布就绪确认、已知问题与规避方案 |

**TR6通过标准**（华为实践）：
- P0（致命）缺陷：0个
- P1（严重）缺陷：0个
- P2（一般）缺陷：关闭率≥95%，遗留的有规避方案
- P3（轻微）缺陷：关闭率≥80%

<!-- chunk: tr-mechanism.tr评审组织与流程 -->
## TR评审组织与流程

### 评审团队（IRT）

| 角色 | 职责 |
|---|---|
| IRT主任 | 主持评审，最终判定通过/不通过 |
| 系统架构师 | 评审架构和接口设计 |
| 各子系统专家 | 评审子系统设计 |
| 质量代表 | 评审质量计划和测试策略 |
| 外部专家 | 必要时邀请（新技术领域） |

### 评审流程

```
PDT提交TR材料（提前5个工作日）
        │
        ↓
IRT成员独立预审（形成预审意见）
        │
        ↓
TR评审会议（2-4小时）
  ├─ PDT技术汇报（30-60分钟）
  ├─ IRT提问与讨论（60-120分钟）
  └─ IRT闭门讨论（15-30分钟）
        │
        ↓
评审结论
  ├─ 通过 → 进入下一阶段
  ├─ 不通过 → 限期整改后重审
  └─ 附条件通过 → 满足条件后进入下一阶段
```

<!-- chunk: tr-mechanism.tr评审的三种结论 -->
## TR评审的三种结论

| 结论 | 含义 | 后续动作 |
|---|---|---|
| **通过** | 技术方案/质量达标 | 进入下一阶段 |
| **不通过** | 存在重大问题 | 限期整改，重新评审 |
| **附条件通过** | 有小问题但可同步整改 | 整改跟踪，不阻塞下一阶段 |

**附条件通过的限制**：仅适用于P2/P3级别问题，P0/P1问题必须不通过。

## TR与质量保证体系

### 缺陷分级标准

| 级别 | 名称 | 定义 | 示例 |
|---|---|---|---|
| P0 | 致命 | 导致系统崩溃或核心功能不可用 | 死机、数据丢失、安全漏洞 |
| P1 | 严重 | 核心功能异常但有 workaround | 性能严重不达标、核心流程中断 |
| P2 | 一般 | 非核心功能异常 | UI缺陷、非核心功能异常 |
| P3 | 轻微 | 不影响功能的体验问题 | 文字错误、界面微调 |

### 缺陷密度目标

| 产品类型 | 目标缺陷密度 |
|---|---|
| 高可靠性产品（通信设备） | <0.5个/KLOC |
| 一般产品（消费电子） | <2个/KLOC |
| 软件产品 | <5个/KLOC |

<!-- chunk: tr-mechanism.常见问题与应对 -->
## 常见问题与应对

| 问题 | 表现 | 应对 |
|---|---|---|
| TR走过场 | 评审不提问题，一律通过 | 设定必检项清单，IRT主任必须提3个以上问题 |
| TR与DCP混为一谈 | 技术评审和商业评审不分 | 明确分离：IRT管TR，IPMT管DCP |
| 评审材料质量差 | 文档来不及写，评审会变成汇报会 | 提前5天提交材料，材料不齐不排评审 |
| 外部专家难请 | 关键技术领域缺专家 | 建立外部专家库，提前预约 |
| TR整改拖沓 | 不通过后迟迟不重审 | 设定整改期限（1-2周），超期升级IPMT |

## 中小企业简化方案

### 三级TR替代六级TR

| 简化TR | 对应华为TR | 评审重点 |
|---|---|---|
| TR-方案评审 | TR1+TR2+TR3 | 方案可行+规格冻结 |
| TR-开发评审 | TR4+TR5 | 开发完成+验证通过 |
| TR-发布评审 | TR6 | 质量达标+发布就绪 |

### 最小化方案（极小团队）
仅保留2个评审点：
1. **TR-立项**：方案评审（对应DCP0前）
2. **TR-上线**：质量评审（对应DCP3前）

<!-- chunk: tr-mechanism.薄云咨询的tr落地建议 -->
## 薄云咨询的TR落地建议

### 落地优先级
1. 先把TR6（发布评审）做实——防止不合格产品上市是底线
2. 再做TR3（规格冻结）——控制需求蔓延的关键点
3. 最后补齐其他TR

### 关键成功因素
- IRT成员必须有技术判断力，不能只是"走流程"
- TR评审必须有预审环节，不能现场翻材料
- TR不通过是正常的、健康的——说明评审在发挥作用
- 每次TR的遗留问题必须追踪到关闭
