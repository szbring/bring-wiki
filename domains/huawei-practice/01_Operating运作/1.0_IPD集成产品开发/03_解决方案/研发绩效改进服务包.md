# 研发绩效改进服务包

<!-- chunk: rd-performance-package.一句话定义 -->
## 一句话定义
基于华为[[IPD]]研发绩效管理实践，帮助企业建立"对商业成功负责"的研发绩效体系，解决研发考核只看产出不看结果、只看个人不看团队的问题。

---

<!-- chunk: rd-performance-package.客户痛点与诊断信号 -->
## 客户痛点与诊断信号

| 痛点 | 表现 | 根因 |
|---|---|---|
| 研发只对"做出来"负责 | 做完了就行，卖不卖得动不管 | 考核指标没有商业维度 |
| 个人绩效与项目脱节 | 研发人员只关心自己模块，不管整体 | 缺少项目级/产品级绩效指标 |
| 考核引发内卷 | 抢容易的任务，推困难的工作 | 指标设计不合理，鼓励短期行为 |
| 跨部门考核难 | 研发和市场的绩效互不相干 | 缺少双轨考核机制 |
| 绩效无法量化 | 评价靠主管感觉 | 缺少量化的绩效指标体系 |

---

<!-- chunk: rd-performance-package.华为研发绩效的核心思想 -->
## 华为研发绩效的核心思想

### 三个转变

| 从 | 到 | 说明 |
|---|---|---|
| 考核个人 | 考核团队+个人 | 产品是团队产出的，个人绩效必须有团队基础 |
| 考核过程 | 考核商业结果 | 研发不是成本中心，是利润中心 |
| 考核技术指标 | 考核商业指标 | 技术指标是过程指标，商业指标是结果指标 |

### 双轨考核模型

```
┌─────────────────────────────────────┐
│           研发人员绩效考核           │
├───────────────┬─────────────────────┤
│   项目绩效     │    部门绩效          │
│   （60-70%）  │    （30-40%）        │
├───────────────┼─────────────────────┤
│ 产品商业结果   │ 专业能力建设         │
│ PDT团队评价   │ 人才培养             │
│ 项目贡献度    │ 技术积累             │
└───────────────┴─────────────────────┘
```

**关键**：项目绩效权重必须>50%，否则人员不会真正投入跨部门项目。

<!-- chunk: rd-performance-package.服务内容 -->
## 服务内容

### SP-RD1：研发绩效快速诊断（3-5人天）

| 内容 | 说明 |
|---|---|
| 现有考核体系审查 | KPI设置、考核流程、结果应用 |
| 关键人员访谈 | 研发骨干/产品经理/部门主管 6-8人 |
| 绩效问题TOP5识别 | 量化分析+根因诊断 |
| 改进方向建议 | 短期速赢+中长期路线图 |

**交付物**：《研发绩效诊断报告》（15-25页）
**报价参考**：8-15万元

---

### SP-RD2：研发绩效体系设计（3-4个月，20-30人天）

#### Phase 1：绩效指标体系设计（4-6周）

**1. 组织级绩效指标**

| 指标类型 | 指标示例 | 权重 |
|---|---|---|
| 财务指标 | 产品线收入、毛利率、研发ROI | 40% |
| 客户指标 | 客户满意度、市占率、NPS | 20% |
| 过程指标 | 研发周期、一次通过率、需求变更率 | 20% |
| 学习成长 | 技术积累、专利、人才培养 | 20% |

**2. PDT团队级绩效指标**

| 指标 | 说明 | 目标值参考 |
|---|---|---|
| 产品商业结果 | 上市后12个月收入vs预测 | ≥90% |
| 按时交付率 | 关键里程碑按计划达成率 | ≥85% |
| 产品质量 | 上市后缺陷率 | <5% |
| 团队协作 | PDT成员互评 | ≥4分/5分 |

**3. 个人绩效指标**

| 角色 | 项目绩效指标 | 部门绩效指标 |
|---|---|---|
| PDT Leader | 产品商业结果+按时交付 | 产品组合规划能力 |
| 研发代表 | 技术方案质量+开发进度 | 技术能力提升+带教 |
| 市场代表 | 需求准确率+上市成功 | 市场洞察能力 |
| 质量代表 | 缺陷发现率+测试覆盖率 | 质量体系建设 |

#### Phase 2：考核流程与制度设计（3-4周）

**1. 考核周期**

| 层级 | 考核周期 | 考核方式 |
|---|---|---|
| 组织级（产品线） | 年度+季度 | 经营分析会评审 |
| PDT团队 | 项目周期（DCP节点） | IPMT评价 |
| 个人 | 季度 | 双轨加权 |

**2. PDT Leader的考核特殊性**
- 由IPMT直接评价（不是功能部门主管）
- 考核核心：产品商业成功（收入/市占/客户满意度）
- 与功能部门主管同级别待遇

**3. 双轨权重分配**

| 人员类型 | 项目绩效权重 | 部门绩效权重 |
|---|---|---|
| PDT核心成员（Leader/研发/市场代表） | 70% | 30% |
| PDT扩展成员（兼职参与） | 50% | 50% |
| 纯职能部门人员 | 20% | 80% |

#### Phase 3：试点运行与优化（4-6周）

- 1-2个PDT试点新绩效方案
- 收集反馈，调整指标和权重
- 与HR系统对接，实现数据化考核

**交付物**：
- 研发绩效指标体系文档
- 各角色绩效考核模板
- 双轨考核制度文件
- 绩效结果应用方案（薪酬/晋升/培训）
- 培训课件

**报价参考**：35-65万元

---

### SP-RD3：研发绩效与IPD联动优化（2个月，8-12人天）

**适用场景**：已有绩效体系，但与IPD流程脱节。

**核心内容**：
1. DCP评审结果与项目绩效的挂钩机制
2. PDT绩效与功能部门绩效的平衡
3. 研发绩效结果在薪酬和晋升中的应用
4. 绩效数据从IPD流程中自动提取

**交付物**：《研发绩效与IPD联动方案》
**报价参考**：18-30万元

<!-- chunk: rd-performance-package.常见误区 -->
## 常见误区

| 误区 | 正确做法 |
|---|---|
| 用代码量考核研发 | 代码量与价值无关，应考核商业结果和代码质量 |
| 考核全部用OKR | OKR适合探索型工作，IPD项目适合KPI+OKR混合 |
| 绩效只挂钩奖金 | 还应挂钩晋升、培训、项目分配 |
| 所有角色同一套指标 | 不同角色（PDT Leader/研发/市场/质量）应有差异化指标 |
| 考核频率过高 | 季度考核即可，月度考核增加管理成本 |

<!-- chunk: rd-performance-package.组合推荐 -->
## 组合推荐

| 客户状态 | 推荐组合 |
|---|---|
| 无IPD框架，绩效问题突出 | SP-RD1（诊断）→ IPD轻量化导入 → SP-RD2（绩效体系） |
| 有IPD框架，绩效与流程脱节 | SP-RD3（联动优化） |
| 预算有限 | SP-RD1（诊断）+ 自实施建议 |
| 全面建设 | IPD深度实施 + SP-RD2（绩效体系） |
