# 需求管理服务包

<!-- chunk: requirement-management-package.一句话定义 -->
## 一句话定义
基于华为OR需求管理方法论，帮助企业建立从需求收集到需求关闭的端到端管理体系，解决"做出来的不是客户要的"问题。

---

<!-- chunk: requirement-management-package.客户痛点与诊断信号 -->
## 客户痛点与诊断信号

| 痛点 | 表现 | 根因 |
|---|---|---|
| 需求混乱 | 需求来源多，格式不统一，无法统一管理 | 缺乏系统化收集渠道 |
| 需求失真 | 客户说的和研发理解的不一样 | 缺乏需求分析和确认机制 |
| 需求膨胀 | 项目中不断加需求，范围蔓延 | 缺乏变更控制和基线管理 |
| 需求遗漏 | 上线后才发现核心需求没做 | 缺乏需求追踪和验证 |
| 需求冲突 | 不同客户的需求互相矛盾 | 缺乏优先级排序和需求仲裁 |
| 需求黑洞 | 需求进去了就看不到状态 | 缺乏全生命周期可视化管理 |

<!-- chunk: requirement-management-package.服务内容 -->
## 服务内容

### SP-OR1：需求管理快速诊断（3-5人天）

| 内容 | 说明 |
|---|---|
| 需求现状访谈 | 6-8位关键人员（产品/研发/市场/销售/客服） |
| 需求流程成熟度评估 | 基于OR成熟度模型打分 |
| 关键需求问题TOP5识别 | 量化分析+根因诊断 |
| 改进建议与路线图 | 优先级排序的改进计划 |

**交付物**：《需求管理诊断报告》（20-30页）
**报价参考**：10-18万元

---

### SP-OR2：需求管理体系建设（3-4个月，15-25人天）

#### Phase 1：需求分类与流程设计（4-6周）

**1. 需求分类体系建立**

基于华为OR分类标准，定制企业需求分类：

| 需求类别 | 符号 | 来源 | 处理方式 |
|---|---|---|---|
| 初始需求 | $ | 客户/市场一线 | 需进一步分析确认 |
| 客户问题 | $$ | 客户投诉/服务反馈 | 需确认是否为真实需求 |
| 内部需求 | I | 研发/运维/管理层 | 需评估对客户的价值 |
| 标准需求 | S | 行业标准/法规 | 必须满足 |
| Bug/缺陷 | B | 测试/客户反馈 | 修复优先 |

**2. 需求处理流程设计**

```
需求收集 → 需求分析 → 需求分配 → 需求实现 → 需求验证 → 需求关闭
   │           │           │           │           │           │
 多渠道      OR-SR       版本规划    开发执行     验证确认     归档
 统一入口    转化分解     优先级排序               可追溯
```

**3. RMT/RAT组织设计**
- RAT（需求分析团队）：一线需求收集和分析
- RMT（需求管理团队）：需求决策和分配

#### Phase 2：工具与模板落地（4-6周）

**1. 需求管理模板包**
- 需求收集表（统一格式）
- 需求分析报告MRS模板
- 需求追踪矩阵RTM模板
- 需求变更申请单
- 需求评审检查单

**2. 需求管理工具选型/配置**
- 支持需求全生命周期管理
- 支持需求追溯（需求→特性→代码→测试）
- 支持需求变更管理
- 与项目管理工具集成

#### Phase 3：试点运行与固化（4-6周）

- 选取1-2个在研项目试点
- 顾问辅导需求分析和管理过程
- 收集反馈，优化流程和模板
- 全量推广

**交付物**：
- 需求管理流程手册
- 需求分类标准文档
- 模板工具包（收集表/MRS/RTM/变更单）
- RMT/RAT运作规范
- 培训课件

**报价参考**：30-60万元

---

### SP-OR3：需求与DCP联动优化（2-3个月，8-15人天）

**适用场景**：已有IPD基础框架，但需求管理与[[IPD]]评审脱节。

**核心内容**：
1. 需求包与Charter的输入关系梳理
2. 需求变更与DCP重新评审的联动机制
3. 需求优先级与产品路标的映射
4. DCP评审中的需求确认检查点

**交付物**：《需求与DCP联动方案》+ 制度文件更新
**报价参考**：20-35万元

<!-- chunk: requirement-management-package.需求管理成熟度模型 -->
## 需求管理成熟度模型

| 级别 | 特征 | 典型表现 |
|---|---|---|
| L1 | 无序 | 需求口头传递，无文档，无追踪 |
| L2 | 已收集 | 有需求文档，但格式不统一，无优先级 |
| L3 | 已分析 | 有分类、有优先级、有基线管理 |
| L4 | 已追踪 | 全链路可追溯，变更受控，质量可度量 |
| L5 | 已优化 | 需求预测、自动分析、持续改进 |

<!-- chunk: requirement-management-package.与ipd标准服务包的关系 -->
## 与IPD标准服务包的关系

```
IPD标准服务包（整体框架）
  ├── Charter/DCP/PDT（立项决策体系）
  ├── 需求管理服务包（本服务包，深化OR部分）
  ├── MM市场管理（深化市场分析部分）
  └── PLM生命周期管理（深化运营部分）
```

**推荐组合**：IPD轻量化导入 + 需求管理服务包 = 立项决策 + 需求管理双轮驱动

---

<!-- chunk: requirement-management-package.组合推荐 -->
## 组合推荐

| 客户状态 | 推荐组合 |
|---|---|
| 无IPD基础，需求混乱 | SP-OR2（需求体系建设）先行，再导入IPD框架 |
| 有IPD框架，需求管理薄弱 | SP-OR3（需求与DCP联动） |
| 需求问题紧急，预算有限 | SP-OR1（快速诊断）+ 自实施建议 |
| 全面建设 | IPD深度实施 + 需求管理服务包 |
