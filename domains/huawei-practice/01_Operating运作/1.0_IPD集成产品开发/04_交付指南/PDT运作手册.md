# PDT运作手册

<!-- chunk: pdt-operations.一句话定义 -->
## 一句话定义
[[IPD]]（Product Development Team）运作手册是PDT团队的日常操作指南，明确PDT的组织、职责、运作规则和关键活动，确保跨功能团队高效协同。

---

<!-- chunk: pdt-operations.pdt定位与使命 -->
## PDT定位与使命

### 定位
PDT是IPD流程的核心执行单元，由跨功能成员组成，对产品从立项到上市的商业成功端到端负责。

### 使命
- 对商业成功负责，而非仅仅"按时做完"
- 跨部门协作内嵌入团队，而非依赖外部协调
- 在预算内按时交付满足市场需求的高质量产品

## PDT组织结构

### 核心团队与扩展团队

```
            PDT Leader
           /    |    \
     研发代表  市场代表  系统工程师
      │        │        │
   扩展团队  扩展团队  扩展团队
   (开发组) (市场组) (架构组)
      │
   采购代表 ── 制造代表 ── 财务代表 ── 质量代表
```

| 类别 | 角色 | 投入度 | 参与方式 |
|---|---|---|---|
| 核心成员 | PDT Leader、研发代表、市场代表、系统工程师 | ≥70% | 全程参与，参加所有核心会议 |
| 扩展成员 | 采购、制造、财务、质量代表 | 30-70% | 关键节点参与，日常代表本职能 |

## PDT各角色职责详述

### PDT Leader

**核心职责**：对产品商业成功端到端负责

| 职责 | 说明 |
|---|---|
| 商业结果 | 产品上市后的收入、利润、市占率 |
| 团队管理 | 组建PDT，分配任务，管理团队绩效 |
| 进度管控 | 确保各阶段按时完成DCP评审 |
| 风险管理 | 识别和管理项目风险 |
| 跨部门协调 | 在PDT内部解决跨部门问题 |
| 汇报 | 定期向IPMT汇报项目进展 |

**PDT Leader的权力**：
- 团队成员的任务分配权
- 团队成员绩效的50%以上打分权
- 项目内资源的调度权
- 跨部门冲突的内部仲裁权

**PDT Leader不是什么**：
- 不是项目经理——项目经理管过程，PDT Leader管商业结果
- 不是协调员——协调员没有实权，PDT Leader有
- 不是部门主管——PDT Leader管项目不管部门

### 研发代表

| 职责 | 说明 |
|---|---|
| 技术方案 | 主导产品技术方案设计 |
| 开发执行 | 组织和推进开发工作 |
| 技术评审 | 参与TR评审，确保技术质量 |
| 技术风险 | 识别和管理技术风险 |

### 市场代表

| 职责 | 说明 |
|---|---|
| 需求输入 | 将市场需求转化为产品需求包 |
| 竞品分析 | 持续跟踪竞争动态 |
| 上市策略 | 制定产品上市和推广计划 |
| 客户验证 | 组织客户验证和反馈收集 |

### 系统工程师

| 职责 | 说明 |
|---|---|
| 系统架构 | 产品系统架构设计 |
| 接口定义 | 子系统接口和协议定义 |
| 技术决策 | 关键技术路径决策 |
| 技术协调 | 跨子系统技术协调 |

### 财务代表

| 职责 | 说明 |
|---|---|
| 预算管理 | 项目预算编制和管控 |
| ROI追踪 | 实际投入vs立项预测对比 |
| 成本分析 | 产品成本结构分析 |
| 财务评审 | DCP财务维度评审支撑 |

### 质量代表

| 职责 | 说明 |
|---|---|
| 质量计划 | 制定项目质量计划 |
| 质量评审 | 参与TR评审，把关质量 |
| 缺陷管理 | 缺陷跟踪和关闭确认 |
| 质量报告 | 定期发布质量状态报告 |

## PDT关键会议体系

### 会议体系总览

| 会议 | 频次 | 参与者 | 时长 | 目的 |
|---|---|---|---|---|
| PDT周例会 | 每周 | 核心成员+相关扩展成员 | 60分钟 | 进度同步、问题决策 |
| DCP评审会 | 按节点 | PDT+IPMT | 2-4小时 | 商业投资决策 |
| TR评审会 | 按节点 | PDT+IRT | 2-4小时 | 技术质量评审 |
| 风险评审会 | 双周 | 核心成员 | 30分钟 | 风险识别和应对 |
| 阶段复盘会 | 每阶段结束 | 全体成员 | 90分钟 | 经验总结、流程改进 |

### PDT周例会议程模板

```
1. 上周行动项回顾（10分钟）
2. 各模块进展汇报（20分钟）
   - 研发：开发进度、技术问题
   - 市场：需求状态、竞品动态
   - 质量：缺陷状态、测试进展
3. 关键问题和决策（20分钟）
4. 下周重点（5分钟）
5. 行动项确认（5分钟）
```

### 会议纪律
- 迟到超过5分钟罚款（团队活动基金）
- 汇报不用PPT，用看板/仪表盘
- 问题必须指定责任人和截止日期
- 会议纪要24小时内发出

## PDT与功能部门的关系

### 双线汇报矩阵

```
                 PDT Leader
                     │
            ┌────────┼────────┐
            │        │        │
        研发代表  市场代表  质量代表
            │        │        │
        研发主管  市场主管  质量主管
```

### 双轨考核权重

| 考核维度 | 考核者 | 权重 |
|---|---|---|
| 项目绩效 | PDT Leader评价 | 60-70% |
| 部门绩效 | 功能部门主管评价 | 30-40% |

### 冲突解决规则

| 冲突类型 | 解决层级 | 规则 |
|---|---|---|
| PDT内部问题 | PDT Leader决策 | 不升级 |
| PDT与部门资源冲突 | PDT Leader与部门主管协商 | 协商不成升级IPMT |
| 多PDT资源冲突 | IPMT协调 | 按战略优先级分配 |
| 技术分歧 | 系统工程师裁决 | 重大分歧IRT评审 |

## PDT常见问题与应对

| 问题 | 根因 | 应对 |
|---|---|---|
| 成员不投入 | 功能部门不放人/不放权 | IPMT强制要求，双轨考核 |
| 会议效率低 | 议题不清，汇报式会议 | 用看板，问题导向 |
| 决策拖延 | PDT Leader不敢决策 | 明确决策权，小问题不过夜 |
| 跨部门推诿 | 责任边界不清 | RACI矩阵明确每项工作的R/A/C/I |
| 需求频繁变更 | 市场代表缺位或需求分析不深 | TR3规格冻结后严格变更控制 |
| PDT Leader被架空 | 无实权，变成协调员 | IPMT授权，绩效打分权 |

## 薄云咨询的PDT辅导方法

### 辅导节奏

| 阶段 | 辅导频次 | 辅导重点 |
|---|---|---|
| 第1-2个PDT | 每周2次 | 手把手教，参与所有核心会议 |
| 第3-4个PDT | 每周1次 | 关注关键节点，其他自行运作 |
| 第5+个PDT | 双周1次 | 抽查式辅导，问题升级时介入 |

### PDT Leader培养
- 选择标准：有商业意识、有领导力、敢做决策
- 培养4步法：影子学习（跟顾问）→ 协助运作（副手）→ 独立运作（顾问观察）→ 教练他人（内部教练）
