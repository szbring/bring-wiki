# 需求跟踪矩阵（RTM）

> 适用阶段：计划阶段（PDCP 前）建立，全程维护至 ADCP
> 使用角色：系统工程师（SE）、产品经理、测试经理
> PDCP 门控要求：RTM 须建立并通过 TR2 评审
> 标签：#模板 #RTM #需求追溯 #TR

---

<!-- chunk: requirement-rtm.一-rtm-的作用 -->
## 一、RTM 的作用

需求跟踪矩阵（Requirement Traceability Matrix）建立从**客户需求到产品实现**的完整追溯链，回答以下问题：
- 每条客户需求有没有被设计到？（正向追溯）
- 每个设计/测试用例对应的是哪条需求？（反向追溯）
- 需求覆盖率是多少？有没有遗漏？

---

<!-- chunk: requirement-rtm.二-追溯链定义 -->
## 二、追溯链定义

```
SF（系统特性）
    ↓
   SR（系统需求）
    ↓
   AR（分配需求/设计规格）
    ↓
   测试用例（TC）
    ↓
   测试结果（Pass/Fail）
```

---

### 字段说明

| 字段 | 说明 |
|------|------|
| **SF 编号/名称** | 系统特性编号，来自 MRS 第三章 |
| **SR 编号** | 系统需求编号，格式 SR-XXX |
| **SR 描述** | 需求规格描述，须可量化/可测试 |
| **AR 编号** | 分配需求编号，格式 AR-XXX |
| **AR 归属模块** | 负责实现该 AR 的子系统/团队 |
| **TC 编号** | 对应测试用例编号 |
| **测试类型** | 单元/集成/系统/性能/环境/客户验证 |
| **测试结果** | Pass ✅ / Fail ❌ / 未测试 — |
| **实现状态** | 已实现 / 开发中 / 未开始 / 取消 |

<!-- chunk: requirement-rtm.四-覆盖率计算方法 -->
## 四、覆盖率计算方法

### 4.1 需求覆盖率（Requirement Coverage Rate）

> 定义：已有测试用例覆盖的 SR 数量 / SR 总数量

```
需求覆盖率 = 有对应 TC 的 SR 数量 ÷ SR 总数量 × 100%

目标：ADCP 前达到 100%（Must 级 SR 必须 100%，Should 级 ≥ 95%）
```

### 4.2 测试通过率（Test Pass Rate）

```
测试通过率 = Pass 的 TC 数量 ÷ 已执行 TC 总数量 × 100%

目标：ADCP 前 Must 级 SR 对应 TC 通过率 = 100%
```

### 4.3 需求实现完成率

```
实现完成率 = 状态为"已实现"的 SR 数量 ÷ SR 总数量 × 100%
```

<!-- chunk: requirement-rtm.六-rtm-维护规则 -->
## 六、RTM 维护规则

| 规则 | 说明 |
|------|------|
| **建立时机** | PDCP 前完成初版，通过 TR2 评审 |
| **更新频率** | 每次需求变更后更新；每周同步测试进展 |
| **变更处理** | 需求变更时，必须同步更新 RTM 中所有关联的 SR/AR/TC |
| **评审节点** | TR2（计划阶段）、TR4A（验证阶段）、ADCP（发布前）|
| **所有权** | 系统工程师（SE）负责维护，PDT Leader 最终审核 |

## 七、反向追溯检查（防止"多余设计"）

除正向追溯外，还需检查：**每个设计模块是否都对应需求？**
（防止工程师自行扩展功能，导致未受控的功能蔓延）

| AR编号 | AR描述 | 对应SR编号 | 是否有需求支撑 |
|--------|--------|----------|--------------|
| AR-001 | | SR-001 | ✅ 有 |
| AR-XXX | 工程师自行添加的功能 X | — | ❌ 无，需提变更申请 |
