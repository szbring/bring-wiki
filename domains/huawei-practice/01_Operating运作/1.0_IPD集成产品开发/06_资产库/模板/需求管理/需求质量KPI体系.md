# 需求质量 KPI 体系

> 适用场景：需求管理过程质量评估、RMT 月度运营、IPD 成熟度提升
> 使用角色：需求管理工程师（REM）、产品经理、质量经理
> 标签：#KPI #需求质量 #度量 #成熟度

---

<!-- chunk: requirement-quality-kpi.一-为什么要度量需求质量 -->
## 一、为什么要度量需求质量

没有度量就没有改进。需求质量差是研发失败的头号原因：
- 需求变更频繁 → 返工 → 延期
- 需求遗漏 → 客户不满意
- 需求过多/无优先级 → 资源浪费

华为 IPD 体系要求在 L3 级别以上建立需求质量度量机制。

## 二、核心 KPI 指标体系

### 2.1 需求稳定性（Requirement Stability）

**定义**：版本计划冻结后，需求的变更频率。变更越少，稳定性越高。

```
需求变更率 = 基线冻结后发生变更的 SR 数量 ÷ SR 基线总数量 × 100%

需求稳定性 = 1 - 需求变更率
```

| 评级 | 需求变更率 | 说明 |
|------|-----------|------|
| 🟢 优秀 | < 5% | 需求管理成熟，变更受控 |
| 🟡 良好 | 5%~15% | 正常范围，需关注变更根因 |
| 🟠 警告 | 15%~30% | 需求分析不充分，需加强 RAT 分析 |
| 🔴 危险 | > 30% | 需求管理失控，严重影响进度和质量 |

**追踪方法**：
- 基线建立后（PDCP 后）开始统计
- 每月更新一次
- 分析变更根因：客户原因 / 内部分析不足 / 市场变化

---

### 2.2 需求覆盖率（Requirement Coverage Rate）

**定义**：已有设计实现（AR）和测试用例（TC）覆盖的 SR 数量比例。

```
需求设计覆盖率 = 已有对应 AR 的 SR 数量 ÷ SR 总数量 × 100%

需求测试覆盖率 = 已有对应 TC 的 SR 数量 ÷ SR 总数量 × 100%
```

| 评级 | 测试覆盖率（Must 级 SR）| 说明 |
|------|----------------------|------|
| 🟢 达标 | 100% | ADCP 通过必要条件 |
| 🟡 待补 | 90%~99% | 需紧急补充 TC |
| 🔴 不达标 | < 90% | ADCP 不应通过 |

**追踪方法**：维护需求跟踪矩阵（RTM），每次 DCP 评审前更新

---

### 2.3 需求召回率（Requirement Recall Rate）

**定义**：产品发布后，客户反馈"缺少某功能"中，**本应在本版本实现但被遗漏**的需求比例。反映需求收集/分析的完整性。

```
需求召回率 = 发布后被客户指出遗漏的 Must/Should 级需求数量 ÷
             本版本 Must/Should 级 SR 总数量 × 100%

（越低越好，目标值：< 2%）
```

| 评级 | 需求召回率 | 说明 |
|------|-----------|------|
| 🟢 优秀 | < 2% | 需求收集和分析质量高 |
| 🟡 良好 | 2%~5% | 需审查 VoC 收集渠道 |
| 🔴 需改进 | > 5% | 需求收集严重不充分 |

**追踪方法**：产品发布后 3 个月收集客户反馈，技术服务团队每月汇总

---

### 2.4 需求质量合格率（Requirement Quality Rate）

**定义**：经 RAT 分析后，描述清晰、可测试、无歧义的 SR 数量占比。

**SR 质量评估标准（SMART 检查）**：
- [ ] **Specific**：描述具体，无歧义
- [ ] **Measurable**：有量化指标（数值/范围/频率）
- [ ] **Achievable**：技术上可实现
- [ ] **Relevant**：与产品目标相关
- [ ] **Testable**：可以编写测试用例验证

```
需求质量合格率 = 满足 SMART 标准的 SR 数量 ÷ SR 总数量 × 100%

目标：≥ 90%
```

---

### 2.5 其他辅助指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| **需求处理周期** | 从 RR 提交到 RMT 决策的平均时间 | ≤ 2 周 |
| **需求积压数量** | 未被 RAT 分析或 RMT 决策的 RR 数量 | < 20 条 |
| **需求接纳率** | 被接纳的 RR 占收集总数比例 | 参考值 40%~60% |
| **CBB 复用率** | 使用已有 CBB 模块满足的 SR 比例 | 目标 ≥ 30% |
| **需求变更成本** | 每次需求变更引起的返工工时 | 持续下降趋势 |

## 四、需求质量改进路径

| 成熟度级别 | 需求质量状态 | 改进重点 |
|------------|------------|---------|
| **L1** | 无度量，靠感觉 | 先建立需求收集表和 RR 登记机制 |
| **L2** | 有简单统计，无分析 | 建立 RTM，开始追踪覆盖率 |
| **L3** | 定期度量，有改进动作 | 全套 KPI 仪表盘，每月 RMT 汇报 |
| **L4** | 数据驱动决策，预测变更趋势 | 历史数据建模，预测需求稳定性风险 |
| **L5** | 持续优化，行业最佳 | 基准对比，主动优化 VoC 收集机制 |
