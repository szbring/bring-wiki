# 需求收集表（VoC 原始需求采集）

> 适用阶段：需求收集阶段（OR 流程第 1 阶段）
> 使用角色：产品经理、销售、技术服务、市场等全体需求收集人员
> 标签：#模板 #需求收集 #VoC #RR

---

<!-- chunk: requirement-collection-sheet.使用说明 -->
## 使用说明

每条原始需求（RR）填写一行。收集后统一汇交 RAT 团队进行分析。
**不要在此阶段过滤或判断需求**——先记录，再分析。

---

<!-- chunk: requirement-collection-sheet.表格字段定义 -->
## 表格字段定义

| 字段 | 说明 | 示例 |
|------|------|------|
| **需求编号** | 格式：RR-YYYYMMDD-序号，如 RR-20260329-001 | RR-20260329-001 |
| **来源渠道** | 客户拜访/销售反馈/竞品分析/招标书/内部反馈/行业标准/技术演进 | 客户拜访 |
| **来源单位/人** | 客户名称（可脱敏）或内部部门 | XX 客户/技术服务部 |
| **来源日期** | 获取该需求的日期 | 2026-03-29 |
| **客户原始描述** | 客户原话或原始反馈，不加工，保留原汁原味 | "我们工厂网络经常断，很影响生产" |
| **需求背景** | 客户面临的业务场景/痛点背景 | 工厂自动化产线对网络稳定性要求极高 |
| **客户类型** | 关键客户/普通客户/潜在客户/内部用户 | 关键客户 |
| **需求频率** | 多个来源反映同一需求的次数（越高优先级越高）| 3 次 |
| **收集人** | 填写本条需求的员工姓名 | 张三 |
| **紧急程度** | 高（客户强烈要求）/中（期望有）/低（偶尔提及）| 高 |
| **初步分类** | 功能需求/非功能需求/约束条件（收集人初判，RAT 可修正）| 非功能需求 |
| **备注** | 其他补充信息，如相关合同条款、竞品是否已有此功能等 | 竞品 A 已支持 |

<!-- chunk: requirement-collection-sheet.appeals-维度覆盖检查 -->
## $APPEALS 维度覆盖检查

在收集需求时，可用以下清单确认 8 个维度均有覆盖：

- [ ] **$（价格）**：客户对价格/成本有无特殊要求？
- [ ] **A（可获得性）**：客户对交货期/供货稳定性有无要求？
- [ ] **P（包装/外观）**：客户对外形/重量/界面有无偏好？
- [ ] **P（性能）**：客户对速度/容量/吞吐量有无指标要求？
- [ ] **E（易用性）**：客户对操作便捷性/学习成本有无关注？
- [ ] **A（可靠性）**：客户对稳定性/质保/认证有无要求？
- [ ] **L（生命周期成本）**：客户关注维护成本/升级成本吗？
- [ ] **S（社会因素）**：客户有环保/国产化/合规要求吗？

<!-- chunk: requirement-collection-sheet.常见需求来源渠道参考 -->
## 常见需求来源渠道参考

| 渠道类型 | 典型来源 |
|----------|---------|
| **客户直接反馈** | 高层拜访、焦点小组、满意度调查、售后投诉 |
| **销售渠道** | 销售机会点记录、客户需求备忘、丢单原因分析 |
| **竞品分析** | 竞品功能对比、行业评测、发布会资料 |
| **标准规范** | 行业标准、国家标准、客户企业标准 |
| **内部反馈** | 技术服务、制造、测试团队的改进建议 |
| **技术演进** | 新技术趋势、专利分析、学术前沿 |

<!-- chunk: requirement-collection-sheet.交接说明 -->
## 交接说明

完成收集后，将本表提交至 **RAT（需求审核团队）**，由 RAT 负责进入分析阶段（过滤→分类→排序→证实），输出 IR 初始需求清单。
