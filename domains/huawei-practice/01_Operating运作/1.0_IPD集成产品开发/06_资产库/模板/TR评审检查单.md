# TR评审检查单

<!-- chunk: tr-checklist.使用说明 -->
## 使用说明
本检查单供IRT成员在TR评审时使用，确保技术评审覆盖所有关键维度。评审前IRT成员应逐项预审PDT提交的技术材料，评审会上逐项确认。

---

<!-- chunk: tr-checklist.tr1-系统方案评审检查单 -->
## TR1 — 系统方案评审检查单

### 架构设计

| # | 检查项 | 通过标准 | 是否达标 | 备注 |
|---|---|---|---|---|
| A1 | 系统架构是否满足产品需求 | 架构覆盖所有核心需求 | ☐ | |
| A2 | 架构是否可扩展 | 预留未来演进空间 | ☐ | |
| A3 | 架构是否合理分层 | 各层职责清晰，耦合度低 | ☐ | |
| A4 | 子系统划分是否合理 | 子系统边界清晰，接口定义 | ☐ | |
| A5 | 是否有架构权衡分析 | 关键架构决策有Trade-off分析 | ☐ | |

### 关键技术

| # | 检查项 | 通过标准 | 是否达标 | 备注 |
|---|---|---|---|---|
| K1 | 关键技术路径是否验证 | POC或仿真验证通过 | ☐ | |
| K2 | 技术风险评估是否完整 | 已识别TOP5技术风险及应对 | ☐ | |
| K3 | 新技术引入是否有评估 | 新技术成熟度、社区支持评估 | ☐ | |
| K4 | 技术选型是否有备选方案 | 关键技术有Plan B | ☐ | |

### 可行性

| # | 检查项 | 通过标准 | 是否达标 | 备注 |
|---|---|---|---|---|
| F1 | 性能目标是否可达 | 有性能估算和验证方法 | ☐ | |
| F2 | 可靠性设计是否考虑 | 有可靠性目标和设计方案 | ☐ | |
| F3 | 安全设计是否考虑 | 有安全威胁分析和防护方案 | ☐ | |
| F4 | 可测试性是否考虑 | 有可测试性设计方案 | ☐ | |
| F5 | 可维护性是否考虑 | 有可维护性设计方案 | ☐ | |

### TR1整体判定

| 结论 | 条件 |
|---|---|
| **通过** | 架构方案可行，关键技术验证通过，风险可控 |
| **不通过** | 架构有重大缺陷或关键技术未验证 |
| **附条件通过** | 有小问题但可同步整改，不影响下一阶段启动 |
