# 问题分类与升级机制

> [[ITR]]流程中对问题进行分级分类，并建立明确的升级路径和ATAC决策机制，确保问题在合适的层级得到高效解决。

---

<!-- chunk: 问题分类与升级机制.一-问题分类体系 -->
## 一、问题分类体系

### 1.1 按来源分类

| 来源 | 说明 | 占比参考 |
|------|------|---------|
| 客户报障 | 客户主动报告的问题 | 60-70% |
| 主动监控 | IoT/监控系统发现 | 15-20% |
| 内部发现 | 巡检/测试中发现 | 10-15% |
| 客户投诉 | 客户不满意升级 | 5-10% |

### 1.2 按类型分类

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 故障类 | 系统不可用或功能异常 | 技术排障→修复 |
| 需求类 | 客户提出新需求 | 评估→进入IPD或LTC |
| 咨询类 | 客户咨询产品使用方法 | 知识库查询→解答 |
| 投诉类 | 客户对产品/服务不满 | 投诉处理流程 |
| 变更类 | 客户需要配置变更 | 评估→执行 |

### 1.3 按影响范围分类

| 范围 | 说明 | 示例 |
|------|------|------|
| 单点 | 仅影响单个用户/设备 | 某台设备故障 |
| 局部 | 影响某个部门/区域 | 某区域网络异常 |
| 系统 | 影响整个系统 | 核心系统宕机 |
| 全局 | 影响所有客户 | 产品通用缺陷 |

<!-- chunk: 问题分类与升级机制.二-问题分级-v-b-a -->
## 二、问题分级（V/B/A）

### 2.1 分级标准

| 级别 | 名称 | 判定标准 | SLA响应 | SLA解决 |
|------|------|---------|---------|---------|
| **V级** | 紧急 | 系统完全不可用，核心业务中断 | 15分钟 | 4小时 |
| **B级** | 严重 | 核心功能受损，业务严重受影响 | 30分钟 | 8小时 |
| **A级** | 一般 | 部分功能异常，业务影响有限 | 2小时 | 24小时 |

### 2.2 V级紧急问题的特征

- 核心系统完全宕机
- 大面积业务中断
- 数据丢失风险
- 安全漏洞被利用
- 影响S级客户

### 2.3 分级调整

问题在处理过程中可能需要调整级别：

| 调整方向 | 触发条件 | 审批 |
|---------|---------|------|
| 升级 | 影响范围扩大或时间超SLA | 一线→二线主管审批 |
| 降级 | 问题影响被高估或已部分缓解 | 二线主管审批 |

## 三、升级机制

### 3.1 三级升级路径

```
一线工程师
  │ → 90%问题在此解决
  │ → 超过SLA时间或无法解决时升级
  ↓
二线专家
  │ → 解决一线升级的疑难问题
  │ → 超过SLA时间或需要产品修改时升级
  ↓
三线研发
  │ → 从产品层面根本解决
  │ → 发布补丁或产品更新
  ↓
问题关闭与知识沉淀
```

### 3.2 升级触发条件

| 升级场景 | 触发条件 | 升级方向 |
|---------|---------|---------|
| 技术升级 | 一线无法解决/需要专业知识 | 一线→二线 |
| 产品升级 | 需要修改产品代码 | 二线→三线 |
| 时效升级 | 超过SLA响应/解决时间 | 自动升级一级 |
| 影响升级 | 影响范围扩大 | 升级问题级别 |
| 客户升级 | 客户主动要求更高级别支持 | 按客户要求升级 |

### 3.3 升级SOP

**一线→二线升级**：
1. 一线工程师判断问题无法独立解决
2. 填写升级工单，附上已排查过程和结论
3. 二线专家15分钟内接单
4. 二线指导一线操作或远程接入处理

**二线→三线升级**：
1. 二线专家判断需要产品修改
2. 填写升级工单，附上根因分析
3. 三线研发评估并制定解决方案
4. 发布热补丁或计划产品更新

**时效升级**：
- 超过SLA响应时间 → 自动升级一级
- 超过SLA解决时间 → 通知上级主管
- 超过SLA 2倍时间 → ATAC决策

## 四、ATAC决策机制

### 4.1 ATAC定义

ATAC（Authorization to Accept Change）是[[ITR]]流程中的关键决策机制，用于处理以下场景：
- 重大问题需要跨部门协调
- 问题解决需要重大资源投入
- 超过SLA时间需要升级决策
- 问题影响范围扩大需要重新评估

### 4.2 ATAC决策场景

| 场景 | 决策内容 | 参与人 |
|------|---------|--------|
| V级问题升级 | 是否启动紧急响应 | 服务总监+研发主管 |
| 超SLA问题 | 是否追加资源或调整方案 | 服务经理+客户经理 |
| 产品缺陷 | 是否发布紧急补丁 | 研发总监+服务总监 |
| 客户投诉升级 | 是否启动客户安抚方案 | 服务总监+AR |

### 4.3 ATAC决策流程

```
问题超SLA/影响扩大 → 发起ATAC → 评估影响和方案 → 决策 → 执行
                                            ↓
                                     需要更高层级 → 升级ATAC
```

### 4.4 ATAC决策要素

| 要素 | 内容 |
|------|------|
| 问题现状 | 当前状态和影响范围 |
| 已采取措施 | 已尝试的解决方案和效果 |
| 推荐方案 | 建议的下一步行动 |
| 资源需求 | 需要投入的人员/设备/预算 |
| 风险评估 | 不采取措施的后果 |
| 时间要求 | 建议的决策时限 |

## 五、问题处理过程管理

### 5.1 过程追踪

| 追踪项 | 内容 | 频率 |
|--------|------|------|
| 问题状态 | 新建/处理中/待确认/已关闭 | 实时 |
| 处理进展 | 当前步骤和下一步 | V/B级每小时 |
| 客户沟通 | 与客户的信息同步 | V级每小时/B级每日 |
| 升级记录 | 升级原因和结果 | 每次升级 |
| 解决方案 | 最终解决方案 | 问题关闭时 |

### 5.2 客户沟通要求

| 问题级别 | 主动沟通频率 | 沟通方式 | 沟通内容 |
|---------|------------|---------|---------|
| V级 | 每小时 | 电话+邮件 | 处理进展、预计恢复时间 |
| B级 | 每日 | 电话/邮件 | 处理进展、解决方案 |
| A级 | 解决后 | 邮件 | 解决方案、预防措施 |

## 六、薄云咨询的应用建议

### 诊断切入点
- 检查是否有问题分级标准和SLA
- 评估升级路径是否畅通（一线→二线→三线）
- 诊断超SLA问题的处理机制

### 辅导重点
- 建立V/B/A分级标准和SLA
- 设计三级升级路径和触发条件
- 建立ATAC决策机制
