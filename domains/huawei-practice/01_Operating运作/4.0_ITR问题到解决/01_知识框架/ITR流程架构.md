# ITR流程架构 — 三阶段与三级服务体系

> ITR（Issue to Resolution，从问题到解决）是华为三大主业务流程之一，实现从客户问题受理到彻底解决并预防再发的端到端价值闭环。

---

<!-- chunk: itr流程架构.一-itr在华为流程体系中的定位 -->
## 一、ITR在华为流程体系中的定位

```
IPD（开发产品）  →  LTC（卖出产品）  →  ITR（解决问题）
     研发              销售               服务
```

华为前常务副总裁费敏总结：**"只有上帝做的东西才没问题。产品当时没问题，时间长了也有问题。问题发生了，就要解决，然后关闭。某代表处的问题解决了叫'解决'，全球此类问题都根治了才叫'关闭'。"**

ITR不是传统的售后流程，而是To Resolution流程——通过客户问题找到解决方案，可能是售后方案，也可能是新销售方案。

<!-- chunk: itr流程架构.二-itr流程架构总览 -->
## 二、ITR流程架构总览

### 2.1 三大主流程

```
┌──────────────────────────────────────────────────────────────┐
│                      ITR 端到端流程                           │
│                                                              │
│  ┌────────────┐   ┌────────────┐   ┌────────────┐           │
│  │ 服务请求   │ → │ 问题处理   │ → │ 问题关闭   │           │
│  │   受理     │   │            │   │            │           │
│  │            │   │            │   │            │           │
│  │ 登记→分级  │   │ 一线→二线  │   │ 验证→回访  │           │
│  │ →派单     │   │ →三线→解决 │   │ →关闭→改进 │           │
│  └────────────┘   └────────────┘   └────────────┘           │
│                                                              │
│  输入：客户问题              输出：问题解决+客户满意+预防再发  │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 ITR的整体框架

ITR流程设计以**管理服务请求**为逻辑起点，**主动维护流程**为关键，**使能流程**为保障：

| 流程类型 | 内容 | 目的 |
|---------|------|------|
| **管理服务请求** | 受理→处理→关闭 | 快速解决客户问题 |
| **主动维护流程** | 巡检→预防→评估 | 预防问题发生 |
| **使能流程** | 升级/备件/返修/回溯 | 支撑复杂问题的解决 |

<!-- chunk: itr流程架构.三-阶段一-服务请求受理 -->
## 三、阶段一：服务请求受理

### 3.1 核心活动

```
客户提单 → 请求登记 → 问题分级 → 派单
```

### 3.2 受理渠道

| 渠道 | 适用场景 | 响应方式 |
|------|---------|---------|
| 服务热线 | 紧急问题 | 即时响应 |
| 官网/APP | 一般问题 | 工单创建 |
| 微信/公众号 | 便捷入口 | 自动创建工单 |
| 邮件 | 非紧急问题 | 工单创建 |
| 现场服务 | 硬件故障 | 派工程师 |
| IoT监控 | 预测性维护 | 主动发起 |

### 3.3 受理信息登记

| 信息项 | 说明 |
|--------|------|
| 客户信息 | 名称、联系人、SLA等级 |
| 问题现象 | 故障描述、影响范围 |
| 环境信息 | 产品型号、版本、配置 |
| 服务历史 | 过往服务记录、已知问题 |
| 保修状态 | 是否在保修期内 |
| 紧急程度 | 根据影响判断优先级 |

### 3.4 分级派单

根据问题严重性和客户SLA等级进行分级派单：

| 问题级别 | 影响程度 | 响应时间 | 处理时限 |
|---------|---------|---------|---------|
| V级（紧急） | 系统完全不可用 | 15分钟 | 4小时 |
| B级（严重） | 核心功能受损 | 30分钟 | 8小时 |
| A级（一般） | 部分功能异常 | 2小时 | 24小时 |

## 四、阶段二：问题处理

### 4.1 三级服务体系

```
┌─────────────────────────────────────────────────────┐
│                  三级服务体系                         │
│                                                     │
│   一线工程师（解决90%常规问题）                       │
│       ↓ 升级                                        │
│   二线专家（解决90%疑难问题）                        │
│       ↓ 升级                                        │
│   三线研发（从根本上解决产品问题）                    │
│                                                     │
│   逐级升级，确保问题在合适层级解决                    │
└─────────────────────────────────────────────────────┘
```

### 4.2 一线服务

**角色定位**：守门人，具备一定技术能力和经验的工程师

**职责**：
- 解决常见问题
- 在技术专家指导下解决疑难问题
- 执行技术专家提供的解决方案
- 查询知识库找到解决方案

**工作方式**：
- 系统弹出案例知识库，供查询
- 能解决→直接出方案指导客户实施
- 不能解决→升级到二线

### 4.3 二线服务

**角色定位**：技术专家，精通特定专业领域，经验丰富

**职责**：
- 处理一线提交的疑难问题
- 提供技术指导
- 协助一线工程师解决复杂问题
- 更新知识库

**工作方式**：
- 远程技术支持
- 指导一线工程师操作
- 必要时到现场支持
- 不能解决→升级到三线

### 4.4 三线服务

**角色定位**：研发团队，从根本的产品层面提供支持

**职责**：
- 解决二线无法通过常规方法解决的问题
- 从根本上调整和优化产品
- 发布热补丁/补丁包
- 预防同类问题再次发生

**工作方式**：
- 深入分析根因
- 发布产品补丁或更新
- 将解决方案反馈到知识库
- 推动产品设计改进

### 4.5 ITR与其他流程的衔接

**ITR→[[LTC]]**：服务过程中识别新订单机会
- 客户需要扩容/升级 → 进入[[LTC]]销售流程
- 客户需要维保续约 → 进入[[LTC]]合同续签

**ITR→[[IPD]]**：挖掘客户需求反馈研发
- 发现产品共性问题 → 输入[[IPD]]需求管理
- 客户提出新功能需求 → 输入[[IPD]]产品规划

## 五、阶段三：问题关闭

### 5.1 关闭流程

```
问题解决确认 → 客户验证 → 客户回访 → 关闭工单 → 知识沉淀
```

### 5.2 关闭标准

- 问题已被有效解决，客户确认
- 客户对解决方案满意
- 相关数据已记录归档
- 解决方案已沉淀到知识库
- 如果是共性问题，已通知其他可能受影响的客户

### 5.3 客户回访

| 回访内容 | 目的 |
|---------|------|
| 问题是否彻底解决 | 确认解决有效性 |
| 服务过程满意度 | 收集服务体验反馈 |
| 是否有新需求 | 发现二次销售机会 |
| 对改进的建议 | 持续改进输入 |

### 5.4 知识沉淀

- 问题现象和根因分析
- 解决方案和步骤
- 预防措施建议
- 关联产品和版本信息
- 贡献到知识库供一线/二线查询

<!-- chunk: itr流程架构.六-主动维护流程 -->
## 六、主动维护流程

### 6.1 主动维护的三个环节

| 环节 | 内容 | 输出 |
|------|------|------|
| 活动规划 | 基于历史数据制定预防性维护计划 | 年度/季度维护计划 |
| 活动实施 | 执行巡检、性能优化、安全加固 | 巡检报告、优化方案 |
| 评估闭环 | 评估维护效果，持续优化 | 维护效果评估报告 |

### 6.2 预防性维护的类型

| 类型 | 频率 | 内容 |
|------|------|------|
| 日常巡检 | 日/周 | 系统健康检查、日志分析 |
| 定期维护 | 月/季 | 性能优化、安全加固、补丁更新 |
| 重大变更前 | 按需 | 变更前评估和备份 |
| 预测性维护 | 持续 | IoT监控+AI预警 |

<!-- chunk: itr流程架构.七-使能流程 -->
## 七、使能流程

| 使能流程 | 内容 | 目的 |
|---------|------|------|
| 紧急恢复流程 | 突发重大故障的应急响应 | 最小化业务中断时间 |
| 问题升级流程 | 问题在当前层级无法解决的升级路径 | 确保问题在合适层级解决 |
| 备品备件流程 | 备件库存管理和调度 | 缩短故障修复时间 |
| 返修流程 | 故障设备送修与返回 | 设备及时恢复可用 |
| 质量回溯和改进 | 问题根因分析和流程改进 | 预防问题再发 |

<!-- chunk: itr流程架构.八-薄云咨询的应用建议 -->
## 八、薄云咨询的应用建议

### 诊断切入点
- 评估客户服务体系是否有三级分层机制
- 检查SLA是否明确且可衡量
- 诊断问题升级路径是否畅通
- 评估知识库的覆盖和质量

### 辅导重点
- 建立三级服务体系和问题升级机制
- 设计SLA标准和考核指标
- 建立知识库和持续改进机制
- 推动ITR→LTC/[[IPD]]的衔接

### 常见误区
- 服务团队没有分层——所有问题都堆给一线
- ITR与LTC/[[IPD]]脱节——服务中发现的机会没有转化
- 只解决不关闭——没有根因分析和预防措施
- 知识库不建设——同样的问题反复出现
