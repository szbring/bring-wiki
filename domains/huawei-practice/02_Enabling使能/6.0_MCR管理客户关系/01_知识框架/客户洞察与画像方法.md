# 客户洞察与画像方法

客户洞察是[[MCR]]流程的起点。华为MCR七大关键点中首要两条——"关注土壤超过庄稼""关注洞察胜于策略"——都指向同一核心：在做客户关系之前，必须先深度理解客户。

---

<!-- chunk: 客户洞察与画像方法.一-客户洞察框架 -->
## 一、客户洞察框架

### 1.1 四维洞察模型

| 维度 | 洞察内容 | 信息来源 | 更新频次 |
|---|---|---|---|
| 行业洞察 | 客户所在行业趋势、政策、竞争格局 | 行业报告、政策文件、展会 | 季度 |
| 企业洞察 | 客户战略、经营状况、组织架构、文化 | 年报、官网、公开信息、交流 | 半年 |
| 决策洞察 | 决策链、权力地图、采购流程、预算机制 | 内部信息、客户交流、历史项目 | 动态更新 |
| 个人洞察 | 关键人的偏好、诉求、风格、痛点 | 日常接触、社交媒体、引荐人 | 持续 |

### 1.2 洞察 vs 信息

| 对比 | 信息 | 洞察 |
|---|---|---|
| 性质 | 事实和数据 | 对事实的深层解读 |
| 价值 | 知道"是什么" | 理解"为什么"和"怎么办" |
| 举例 | 客户CIO姓张，50岁 | 张CIO即将退休，正为退休后安排焦虑，需要稳定业绩证明 |
| 获取方式 | 公开渠道、CRM系统 | 深度交流、分析推理、交叉验证 |

## 二、客户决策链分析

### 2.1 B2B决策链角色模型

```
        ┌─────────────┐
        │   决策者      │  最终签字人（CEO/VP）
        │  Decision Maker│
        └──────┬──────┘
               │
     ┌─────────┴─────────┐
     │                    │
┌────┴────┐        ┌────┴────┐
│ 评估者   │        │ 把关者   │  采购/合规/法务
│ Evaluator│        │ Gatekeeper│
└────┬────┘        └────┬────┘
     │                    │
     └─────────┬─────────┘
               │
        ┌──────┴──────┐
        │   使用者      │  实际使用产品的人
        │   End User   │
        └─────────────┘
```

### 2.2 六类关键角色

| 角色 | 英文 | 定义 | 关注点 | 应对策略 |
|---|---|---|---|---|
| 决策者 | Decision Maker | 最终拍板人 | 投资回报、战略匹配、风险控制 | 高层对接，讲价值讲战略 |
| 评估者 | Evaluator | 方案技术评估 | 功能、性能、技术指标 | 技术团队对接，深度技术交流 |
| 把关者 | Gatekeeper | 流程合规审核 | 合规、预算、流程 | 提前沟通，确保流程合规 |
| 使用者 | End User | 实际使用者 | 易用性、效率提升、培训 | 重视体验，做好培训和售后 |
| 影响者 | Influencer | 有话语权但无决策权 | 行业趋势、同行实践 | 借助行业活动建立影响 |
| 教练 | Coach | 内部支持者 | 帮你赢就是帮他赢 | 深度绑定，全力支持 |

### 2.3 决策链分析方法

1. **组织架构分析**：从客户公开组织架构入手，找到各部门负责人
2. **项目历史回溯**：分析过往项目的决策路径，谁说了算
3. **信息交叉验证**：从不同渠道（销售、交付、客户交流）收集信息比对
4. **动态更新**：每次客户接触后更新决策链认知

## 三、权力地图

### 3.1 权力地图定义

权力地图是客户内部权力分布的可视化工具，不仅展示正式的组织权力，还揭示非正式的影响力网络。

### 3.2 权力地图绘制方法

| 步骤 | 动作 | 输出 |
|---|---|---|
| 第一步 | 绘制正式组织架构图 | 明确汇报关系和正式权力 |
| 第二步 | 标注项目相关人及其角色 | 决策者/评估者/把关者/使用者 |
| 第三步 | 标注与我方关系状态 | Coach/Supporter/Neutral/Blocker |
| 第四步 | 标注非正式影响力线 | 谁听谁的、谁影响谁 |
| 第五步 | 标注竞争格局 | 谁支持竞品、谁倾向我方 |

### 3.3 权力地图模板

```
客户权力地图

Level 1:  CEO ─────────── [中立] ──────── 最终决策
          │
Level 2:  CTO ──── [支持者] ──── 技术评估    CFO ──── [把关者] ──── 预算审核
          │                              │
Level 3:  IT总监 ── [教练] ── 方案推荐     采购总监 ── [中立偏反] ── 采购流程
          │                              │
Level 4:  架构师 ── [支持者] ── 技术评分     采购专员 ── [反对者] ── 执行采购

非正式影响力：
CTO ←→ CEO（每周高尔夫）
IT总监 → CTO（前下属关系）
架构师 → IT总监（技术权威）
```

### 3.4 使用场景

- 项目投标前：看清决策链，制定公关策略
- 客户关系规划：识别薄弱环节，分配资源
- 竞争态势分析：识别竞品的支持者和反对者
- 新客户开拓：快速建立对客户决策机制的理解

## 四、客户价值评估

### 4.1 B2B客户价值评估三维度

| 维度 | 评估内容 | 指标 | 权重 |
|---|---|---|---|
| 当前价值 | 客户当前贡献 | 营收、利润、份额 | 40% |
| 战略价值 | 客户的战略意义 | 品牌背书、行业标杆、技术引领 | 30% |
| 成长价值 | 客户未来潜力 | 行业增长率、客户自身增长、交叉销售机会 | 30% |

### 4.2 客户价值评分卡

| 评分项 | 1分 | 3分 | 5分 |
|---|---|---|---|
| 年营收贡献 | <100万 | 100-500万 | >500万 |
| 利润率 | <10% | 10-20% | >20% |
| 品牌价值 | 无 | 行业内有一定知名度 | 行业头部，品牌背书效应强 |
| 成长性 | 行业萎缩 | 行业平稳增长 | 行业高速增长 |
| 交叉销售潜力 | 仅1条产品线 | 2-3条产品线 | 5条以上产品线 |
| 合作稳定性 | 每年重新招标 | 多年合作，偶有竞争 | 长期战略合作 |
| 付款信用 | 经常逾期 | 偶尔逾期 | 从不逾期 |

### 4.3 客户分级标准

| 级别 | 定义 | 评分区间 | 投入策略 |
|---|---|---|---|
| S类 | 战略客户 | 30-35分 | 最高优先，高层直管 |
| A类 | 伙伴客户 | 22-29分 | 重点投入，客户经理专职 |
| B类 | 普通客户 | 15-21分 | 标准服务，定期维护 |
| C类 | 长尾客户 | <15分 | 标准化服务，低接触 |

## 五、客户画像应用

### 5.1 客户画像要素

| 要素类别 | 具体内容 |
|---|---|
| 基本信息 | 企业名称、行业、规模、所在地、成立年限 |
| 经营信息 | 营收、利润、增长率、主要客户、竞争对手 |
| 组织信息 | 组织架构、决策机制、采购流程、预算周期 |
| 关键人信息 | 决策者、影响者、对接人、其偏好和诉求 |
| 合作信息 | 合作历史、当前项目、待续约项目、竞品渗透 |
| 价值评估 | 当前价值、战略价值、成长价值、综合评级 |

### 5.2 画像更新机制

| 事件 | 触发更新 |
|---|---|
| 客户高层变动 | 立即更新决策链和权力地图 |
| 重大项目中标/失标 | 更新合作信息和竞品渗透 |
| 季度业务回顾 | 更新经营信息和合作进展 |
| 行业重大变化 | 更新行业洞察和客户战略 |
| 年度客户规划 | 全面更新客户画像 |
