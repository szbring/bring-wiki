## 一、核心理念：什么是好的变革管理

华为25年变革实践提炼出三个核心主张：

> 1. **变革是最高领导力** — 变革领导力是一把手工程，赞助人级别决定变革成败
> 2. **先僵化后优化再固化** — 不要在引进之初就改动，先原版学习，再本地化优化，最后制度固化
> 3. **变革的本质是改变人** — 流程和系统是手段，人的观念、意识和行为改变才是变革的真正目的

<!-- chunk: 华为变革管理体系精要.二-先僵化后优化再固化 -->
## 二、"先僵化后优化再固化"

### 2.1 原则出处

1999年华为引入IBM的[[IPD]]体系时，任正非明确提出这一变革推行原则。

### 2.2 三阶段含义

| 阶段 | 要求 | 时间 | 常见误区 |
|------|------|------|---------|
| **先僵化** | 原版引入，不随意改动 | 前1-2年 | 没学会就开始"改良"，结果两不像 |
| **后优化** | 在运作中发现问题，结合实际本地化 | 2-4年 | 把"优化"变成了推翻重来 |
| **再固化** | 优化成果变成制度和文化，不可逆转 | 4年以后 | 固化流于形式，无人执行 |

### 2.3 背后逻辑

"僵化"不是贬义词，而是**保持敬畏、不急于求成**的态度：
- IBM的[[IPD]]体系是经过数十年实践检验的体系，贸然改动极可能丢掉精华
- 先"僵化"运作一段时间，才能基于真实经验判断哪里需要优化
- 变革成功后的"固化"，是让组织能力成为不依赖个人的"无生命体系"

### 2.4 咨询应用

薄云咨询在[[IPD]]/HR/[[LTC]]等体系导入时，均遵循这一原则：
- **第一期（6-12个月）**：照搬华为模板，建立基本运作规范
- **第二期（12-24个月）**：根据业务实际优化关键流程，调整不适配的内容
- **第三期（持续）**：将成功实践写入制度、嵌入IT系统、纳入绩效考核

## 三、变革管理"船模型"（8要素框架）

### 3.1 模型概述

"船模型"是华为变革管理最核心的操作框架。将变革比作一艘船渡河，8个要素构成船体不同部分，缺一不可。

```
      ╔══════════════════════════════════════════════╗
      ║              赞助人（船长）                    ║
      ║  ——看得见、站在船头、调配资源、扫清障碍——      ║
      ╚══════════════════════════════════════════════╝
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  ① 发展赞助人   ② 项目组发展    ③ 利益关系人管理          │
  │                                                          │
  │  ④ 组织文化调整 ⑤ 组织和职位设计 ⑥ 绩效管理激励          │
  │                                                          │
  │  ⑦ 沟通        ⑧ 教育和培训                              │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
                 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
                         业务目标（彼岸）
```

### 3.2 八要素详解

**① 发展赞助人（Sponsor Development）**

变革成功的**第一要素**。赞助人的核心行为是"拉绳子"而非"推绳子"——主动拉动变革向前，而不是在后面推。

| 合格赞助人行为 | 不合格赞助人行为 |
|-------------|--------------|
| 公开、频繁表态支持 | 只在启动会露面，之后消失 |
| 主动清除障碍，调配资源 | 遇到阻力让项目组自己解决 |
| 在高管团队内为变革背书 | 在高管会议中对变革态度暧昧 |
| 直接与关键反对者沟通 | 把所有冲突都甩给项目经理 |

**② 项目组发展（Team Development）**

变革需要专职团队，不能完全靠兼职推进：
- 核心成员必须有**80%+时间**投入变革项目
- 团队需要变革管理知识 + 业务知识的复合型人才
- 项目组成员是未来组织的"变革种子"

**③ 利益关系人管理（Stakeholder Management）**

用"意愿×能力"四象限分类管理：

```
        高支持度
            │
 能力弱+意愿强   ║  能力强+意愿强
 （培养盟友）   ║  （核心盟友，重点依靠）
 ─────────────────────────────────→ 能力
 能力弱+意愿弱   ║  能力强+意愿弱
 （边缘观望）   ║  （潜在阻力，需要转化）
            │
        低支持度
```

**④ 组织文化调整（Culture Alignment）**

变革往往与现有文化存在冲突，需要有意识地：
- 树立榜样：让文化先行者在变革中获得荣誉和奖励
- 讲故事：用真实案例传播"变革是正确的"信念
- 仪式化：里程碑庆祝、变革英雄表彰

**⑤ 组织和职位设计（Organization & Role Design）**

流程变了，组织架构必须同步调整，否则"新瓶装旧酒"：
- 新流程 → 重新定义岗位职责
- 新岗位 → 配套汇报关系和授权机制
- 新机制 → 更新岗位说明书和能力模型

**⑥ 绩效管理激励（Performance & Incentive）**

> "你考核什么，你就得到什么。"

变革期间绩效激励设计的关键：
- 在新行为刚开始时给予**超额激励**，帮助建立习惯
- 消除反向激励：不能一边要求新行为，一边按旧KPI考核
- 短期指标（行为指标）+ 长期指标（结果指标）结合

**⑦ 沟通（Communication）**

变革沟通三原则：
1. **宁多勿少**：关键信息需要多渠道、多频次重复
2. **双向沟通**：不只是发布信息，更要倾听反馈和焦虑
3. **针对受众**：CEO关心为什么变，中层关心对自己的影响，基层关心怎么做

沟通三件事：
- **Burning Platform**（燃烧的平台）：让大家看到不变革的后果
- **Vision**（愿景蓝图）：让大家看到变革成功后的美好图景
- **What's In It For Me**（对我有什么好处）：每个人都需要个性化的理由

**⑧ 教育和培训（Education & Training）**

区分两类培训需求：
- **知识培训**（Know-how）：让人知道新流程/系统怎么用
- **行为培训**（Behavior Change）：让人真正改变工作方式

华为的实践：培训必须与变革推行节奏严格同步，"上线前培训 → 上线后辅导 → 运行中强化"三阶段设计。

## 四、七个反对

> 2006年，华为在大规模变革推进中总结出"七个反对"，是防止变革走偏的红线原则。

| 序号 | 反对内容 | 核心意图 |
|------|---------|---------|
| **① 反对完美主义** | 不要等到流程完美了再推行 | 先粗后细，在运作中优化。完美是好的敌人 |
| **② 反对繁琐哲学** | 流程不能越来越复杂 | 简洁有效才是好流程，避免官僚化 |
| **③ 反对盲目创新** | 不能为创新而创新 | 要基于业务价值，不是为了标新立异 |
| **④ 反对没有全局效益提升的局部优化** | 局部最优≠全局最优 | 变革必须看系统效益，不能只优化本部门 |
| **⑤ 反对没有全局观的干部主导变革** | 视野局限的干部会把变革导向局部利益 | 变革领导者必须有跨部门全局视野 |
| **⑥ 反对没有业务实践经验的人参加变革** | 纯理论派做出的流程无法落地 | 变革团队必须有真正懂业务的人 |
| **⑦ 反对未经充分论证的流程进入实用** | 大规模推广前必须经过试点验证 | 先试点，再推广，避免全面翻车 |

## 五、变革管理8阶段流程

```
变革规划 → 概念 → 计划 → 开发 → 验证 → 试点 → 推行 → 运营管理
    │         │       │       │       │       │       │       │
 战略对齐   Charter  详细   原型/   小范围  受控   全面   固化+
 项目识别  立项批准  计划   系统    验证   推广   推广   持续优化
```

| 阶段 | 核心任务 | 关键决策门（DCP） | 常见风险 |
|------|---------|----------------|---------|
| **变革规划** | BT战略对齐，识别变革机会 | — | 变革项目与战略脱节 |
| **概念** | 明确范围、目标、方法，编写Charter | DCP0：立项批准 | Charter太模糊，目标不可量化 |
| **计划** | 详细计划、资源、风险管理 | DCP1：计划批准 | 资源不足，里程碑不现实 |
| **开发** | 流程设计、系统开发、培训材料 | DCP2：开发完成 | 流程设计脱离业务实际 |
| **验证** | 受控环境验证新流程/系统 | DCP3：验证通过 | 验证范围太窄，遗漏关键场景 |
| **试点** | 1-3个单位受控试点 | DCP4：试点评估 | 试点结果无代表性，过于顺利或过于艰难 |
| **推行** | 全面推广 | DCP5：推行完成 | 推广节奏过快，支撑资源不足 |
| **运营管理** | 固化、优化、绩效监控 | — | 项目一结束，就没人管了 |

## 六、ESC：变革最高决策机构

**ESC（Enterprise Steering Committee，企业变革指导委员会）** 是华为变革管理的最高治理机构。

**职能**：
- 审批、优先排序全公司变革项目（防止各自为政，项目泛滥）
- 统一管理变革预算（2014年后，变革预算统一由ESC管理）
- 协调跨领域资源冲突
- 公司最高级别的变革赞助人群体

**郭平的ESC工作实践（2014-2022年任主任）**：
- 每周定期召开ESC会议（不能因为忙就取消）
- 亲自推动重大变革项目的关键决策
- 对阻力最大的变革亲自出面背书

## 七、变革治理完整架构

```
ESC（企业指导委员会）—— 最高决策，统管所有变革
├── EAC（企业架构委员会）—— 4A架构统一管理，防止"烟囱"
├── 领域3T（变革总监 + 流程Owner + IT Owner）—— 各领域端到端推进
│   ├── 赞助人（Sponsor）—— 业务高层背书，资源支持
│   └── 变革项目团队 —— 执行层
└── PO（流程Owner委员会）—— 流程架构统一规范
```

## 八、薄云咨询的方法论选择

| 核心框架 | 来源 | 主要应用场景 |
|---------|------|------------|

| 七个反对 | 华为总结 | 变革推进红线管理 |
| ADKAR模型 | Prosci | 个体层面变革管理 |
| 变革之桥（Bridges） | Bridges | 转型期人员情绪管理 |
