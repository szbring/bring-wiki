# 企业架构EA方法论

<!-- chunk: 企业架构ea方法论.一句话定义 -->
## 一句话定义
华为"一体四面"企业架构（4A），以业务架构为锚点，数据架构全局拉通，应用架构合理呈现，技术架构有效支撑，确保业务与IT从战略到落地的一致性。

---

<!-- chunk: 企业架构ea方法论.一-华为ea发展历程 -->
## 一、华为EA发展历程

| 时间 | 里程碑 |
|---|---|
| 2008 | 启动EA项目，发布EA Framework 1.0 |
| 2009 | 发布AA 1.0、TA 1.0，成立EAC和SAG团队 |
| 2010 | 发布BPA 2.0及配套IA/AA/TA 2.0 |
| 2012 | 借鉴TOGAF启动EA Method 2.0，ARIS架构建模 |
| 2013 | EA方法融入变革与IT规划流程，实现集团到BG/区域分层 |
| 至今 | 持续演进，支撑数字化转型和智能化升级 |

<!-- chunk: 企业架构ea方法论.二-4a架构框架 -->
## 二、4A架构框架

### 华为"一体四面"

```
┌────────────────────────────────────────────┐
│            一体：瞄准业务目标                │
│    架构师团队协同进行框架设计                │
├──────────┬──────────┬──────────┬───────────┤
│ 业务架构  │ 数据架构  │ 应用架构  │ 技术架构   │
│  (BA)    │  (DA)    │  (AA)    │  (TA)     │
│ 整体牵头  │ 全局拉通  │ 合理呈现  │ 有效支撑   │
└──────────┴──────────┴──────────┴───────────┘
```

### 四面协作关系

| 架构 | 角色 | 核心任务 | 关键产出 |
|---|---|---|---|
| BA 业务架构 | 整体牵头 | 构建价值链-流程-能力-业务对象 | 业务流程架构(BPA)、业务能力地图 |
| DA 数据架构 | 全局拉通 | 构建企业级数据资产体系 | 数据资产目录、数据标准、主数据模型 |
| AA 应用架构 | 合理呈现 | 线上化呈现业务活动 | 应用系统蓝图、功能模型、集成关系 |
| TA 技术架构 | 有效支撑 | 提供运行基础设施和平台 | 技术框架、组件、部署架构 |

**核心原则**：所有架构围绕"业务对象"设计（如产品、客户、合同、订单），确保4A整体协同。

<!-- chunk: 企业架构ea方法论.三-业务架构-ba -->
## 三、业务架构（BA）

业务架构是4A的起点和锚点，回答三个根本问题：
1. 企业如何创造价值？
2. 企业如何组织业务流程与活动？
3. 企业需要哪些能力来支撑战略目标？

| 要素 | 说明 |
|---|---|
| 价值流 | 端到端价值创造路径（如[[IPD]]、LTC、ITR） |
| 业务能力 | 支撑价值流所需的能力（可引入数字技术重构） |
| 业务流程 | 能力的流程化表达（L1-L6层级） |
| 业务对象 | 流程操作的核心实体（产品、客户、合同等） |
| 组织角色 | 流程的执行者和责任人 |

---

<!-- chunk: 企业架构ea方法论.四-数据架构-da -->
## 四、数据架构（DA）

数据架构不是数据库设计，而是从全企业视角构建数据资产体系：

| 要素 | 说明 |
|---|---|
| 数据资产目录 | 数据世界的分类法，按业务域分层数据资产 |
| 数据标准 | 统一"数据语言"，消除语义不一致 |
| 主数据管理 | 跨域共享的核心数据（客户、产品、组织等） |
| 数据模型 | 概念模型、逻辑模型、物理模型三层设计 |
| 数据分布 | 数据在系统间的分布与流转路径 |

**华为数据治理"4层保障+2个抓手"**：
- 4层保障：政策保障、组织保障、流程保障、IT落地保障
- 2个抓手：信息架构、数据质量

<!-- chunk: 企业架构ea方法论.五-应用架构-aa -->
## 五、应用架构（AA）

| 要素 | 说明 |
|---|---|
| 应用系统蓝图 | 企业应用系统全景及目标架构 |
| 功能模型 | 功能项-功能子项分解，形成能力矩阵 |
| 集成关系 | 系统间接口、数据流、事件流 |
| 服务目录 | 可复用的应用服务（查询、校验、主数据等） |
| 现状映射 | 现有系统到目标架构的差距与路线图 |

### 应用分层

```
前台应用 ── 用户体验、业务编排
  │
中台应用 ── 共享服务、业务规则
  │
后台应用 ── 核心交易、数据处理
  │
基础平台 ── 技术组件、基础设施
```

## 六、技术架构（TA）

| 层次 | 内容 |
|---|---|
| 策略 | 技术战略、原则、标准 |
| 框架 | 终端接入、网络联接、数据/AI平台、集成中间件、运行环境、安全运维 |
| 组件 | 数据库、消息系统、API网关、缓存、计算引擎等 |
| 服务 | 数据服务、AI服务、集成服务等 |
| 平台 | 数据中台、AI平台、集成平台、运行管理平台、云平台 |
| 部署 | 上云策略(6R)、混合云、容器化、边端协同 |

<!-- chunk: 企业架构ea方法论.七-三层架构治理 -->
## 七、三层架构治理

华为EA分为三个层面：

| 层面 | 范围 | 治理机制 |
|---|---|---|
| 企业层（Corporate-wide） | 集团级通用架构资产 | EAC审批，SAG指导 |
| 领域层（Business Area/BG） | BG/区域级架构 | 领域架构师设计 |
| 解决方案层（Solution） | 项目级架构 | 方案架构师设计，EAC合规审查 |

### 架构治理组织

| 组织 | 职责 |
|---|---|
| EAC（企业架构委员会） | 颁布架构原则和内容框架，审批架构蓝图 |
| SAG（资深架构专家组） | 细化领域架构原则，指导架构师 |
| RAMT（需求与架构管理团队） | 4A架构师组合，对接变革项目 |
| 3T（变革与IT管理团队） | 业务变革与IT实施 |

<!-- chunk: 企业架构ea方法论.八-与togaf的关系 -->
## 八、与TOGAF的关系

华为EA方法借鉴了TOGAF ADM（架构开发方法）的8阶段流程：

| TOGAF阶段 | 华为对应实践 |
|---|---|
| A.架构愿景 | 战略解码→EA愿景 |
| B.业务架构 | BPA L1-L3设计 |
| C.信息系统架构 | IA+AA设计 |
| D.技术架构 | TA设计 |
| E.机会及解决方案 | 差距分析→方案设计 |
| F.迁移规划 | 变革项目路标 |
| G.实施治理 | 架构合规审查 |
| H.架构变更管理 | 架构持续演进 |
