# 业务变革管理（BT）

## 定义
通过有组织、有方法、有资源保障的方式，把战略诉求转化为流程、组织、IT、文化的协同变革，并经由 BTC 治理与变革 DCP 评审实现端到端落地。

## 输入 / 输出
- **输入**：战略变革诉求（来自 DSTE）+ 业务痛点 + 外部对标 + 资源约束
- **输出**：变革 charter（立项书）→ 变革方案 → 变革落地 → 沉淀的新能力（流程文件 + 组织调整 + 系统 + 文化）
- **关键里程碑**：立项评审（Charter DCP）/ 方案评审 / 试点评审 / 推广评审 / 关闭评审

## 角色
- **指导方**：BTC（业务变革指导委员会）
- **执行方**：变革项目 PMO + 业务专家 + 流程专家 + IT 顾问
- **共担方**：业务流 Owner + 受影响部门一把手
- **变革代理**：被识别出的"变革引导者"群体

## 上下游
- **上游**：DSTE 战略 / 业务痛点 / 流程评估发现的差距
- **下游**：流程更新 → 各业务流 Owner 接管运行；IT 落地 → IT 部门维护
- **关键关口**：变革 Charter 评审、各阶段 DCP（参考 [[id:huawei.overview.concept.dcp]]）

## 典型误区
- **变革无 Charter**：项目目标、范围、资源、责任人不清，最终扯皮
- **变革不动组织**：流程改了但 KPI 和组织架构未配套，新流程被旧考核绞杀
- **试点即推广**：试点成功就全面铺开，不评估边界条件——失败率高
- **变革办公室成永久组织**：变革完成后 PMO 不解散，反而养成新部门——能力没沉淀到业务
