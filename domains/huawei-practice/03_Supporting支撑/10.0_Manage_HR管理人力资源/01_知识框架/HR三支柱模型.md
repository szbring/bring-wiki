# HR三支柱模型

<!-- chunk: hr三支柱模型.模型概述 -->
## 模型概述

```
        ┌─────────────┐
        │    COE      │  专家中心（政策与标准制定）
        │  专家中心   │
        └──────┬──────┘
               │ 政策/工具/方法论
    ┌──────────┼──────────┐
    │          │          │
┌───▼───┐  ┌──▼───┐  ┌───▼───┐
│ HRBP  │  │ HRBP │  │ HRBP  │  业务伙伴（嵌入各BU）
│ 研发  │  │ 销售 │  │ 职能  │
└───────┘  └──────┘  └───────┘
               │
        ┌──────▼──────┐
        │    SSC      │  共享服务中心（标准化事务处理）
        │  共享服务   │
        └─────────────┘
```

## 三支柱职责

### HRBP — HR Business Partner（业务伙伴）

**核心定位**：懂业务的HR，而不是懂HR的业务

**主要职责**：
- 深入理解所支持业务单元的战略和挑战
- 参与业务决策（尤其是组织和人才相关决策）
- 人才盘点：识别关键岗位和关键人才，制定人才发展计划
- 组织健康度诊断：发现组织文化和效能问题
- 变革管理：协助业务部门推进管理变革

**华为的「政委」文化**：
- HRBP像部队中的政委，与业务负责人是搭档关系
- 有权对干部提出任免建议
- 任正非名言：「谁是最好的HR？赵刚（李云龙的政委）」

**与传统HR的区别**：
| 传统HR | HRBP |
|---|---|
| 等业务找我 | 主动融入业务 |
| 执行HR制度 | 参与业务决策 |
| 管理人事事务 | 解决人才问题 |
| 关注合规 | 关注业务结果 |

---

### COE — Center of Expertise（专家中心）

**核心定位**：HR领域的专家，制定政策、工具和最佳实践

**主要职责**：
- 任职资格标准制定与更新
- 薪酬政策和结构设计
- 绩效管理体系设计
- 培训与发展体系建设
- 招聘标准和流程设计
- HR数字化工具选型和建设

**典型COE模块**：
- 薪酬福利COE
- 人才发展COE
- 招聘COE
- 组织发展COE

---

### SSC — Shared Service Center（共享服务中心）

**核心定位**：高效处理标准化HR事务，让HRBP从行政中解放出来

**主要职责**：
- 入职/离职手续办理
- 考勤与假期管理
- 薪酬核算与发放
- 社保公积金办理
- HR系统维护
- 员工查询和咨询服务（标准问题）

<!-- chunk: hr三支柱模型.三支柱转型路径 -->
## 三支柱转型路径

### 适合导入三支柱的条件
- 员工规模 ≥ 200人（太小ROI低）
- 业务多元化（有多个业务单元/部门）
- HR希望从「事务型」向「战略型」转型
- 高层支持HR转型

### 导入步骤（3-6个月）

**Step 1：现状诊断**（1个月）
- HR现有工作内容分类（事务性/政策性/业务支持）
- 确认SSC可承接的标准化事务清单
- 识别哪些业务单元最需要HRBP支持

**Step 2：COE和SSC建立**（1-2个月）
- SSC：整理标准化流程，建立操作手册和FAQ
- COE：梳理现有HR政策，确认各模块负责人

**Step 3：HRBP配置**（1-2个月）
- 按业务单元配置HRBP（每100-150人1个HRBP）
- HRBP能力培训：业务理解+组织诊断+数据分析
- 与业务负责人建立伙伴关系

**Step 4：持续优化**（持续）
- HRBP定期与业务负责人对齐
- COE政策根据业务反馈迭代
- SSC自动化程度持续提升

<!-- chunk: hr三支柱模型.常见问题 -->
## 常见问题

| 问题 | 解决方案 |
|---|---|
| HRBP不被业务接受 | 先找1-2个「胜仗」，用结果证明价值 |
| COE脱离实际 | 每季度与HRBP对齐，确保政策可执行 |
| SSC服务质量差 | 建立服务SLA，定期考核响应时间和准确率 |
| 三支柱互相推诿 | 建立清晰的RACI，明确边界 |
