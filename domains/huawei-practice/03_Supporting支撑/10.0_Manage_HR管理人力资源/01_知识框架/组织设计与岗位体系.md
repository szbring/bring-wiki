# 组织设计与岗位体系

<!-- chunk: 组织设计与岗位体系.核心价值 -->
## 核心价值

> **「组织是为了战略服务的，不是战略迁就组织」** — 组织设计从战略出发，岗位体系是组织运转的最小单元，也是人力资源管理的基石。

---

<!-- chunk: 组织设计与岗位体系.华为组织设计的基本原则 -->
## 华为组织设计的基本原则

### 战略决定组织

```
战略方向 → 业务设计 → 组织架构 → 岗位设置 → 人员配置
```

华为每次战略调整都伴随组织变革：
- 从单产品到多产品：成立产品线组织
- 从国内到海外：建立区域组织
- 从运营商到消费者到企业：设立三大BG（Business Group）

### 组织设计的五条原则

| 原则 | 内涵 | 反面教训 |
|---|---|---|
| 客户导向 | 组织围绕客户设置，不是围绕领导 | 内部导向，远离客户 |
| 扁平高效 | 减少管理层级，提升决策效率 | 层层汇报，错失战机 |
| 前端灵活 | 一线有权呼唤炮火 | 前线请示后方，反应迟钝 |
| 后端平台 | 能力共享，避免重复建设 | 每个部门自建小而全 |
| 动态调整 | 战略变则组织变 | 组织僵化，路径依赖 |

<!-- chunk: 组织设计与岗位体系.华为组织架构的演变 -->
## 华为组织架构的演变

| 阶段 | 组织形态 | 特点 |
|---|---|---|
| 早期（1988-1998） | 职能制 | 按专业分工（研发/销售/生产），简单直接 |
| 成长期（1998-2005） | 矩阵制 | 引入[[IPD]]后，产品线+职能线双维管理 |
| 扩张期（2005-2012） | BG制 | 运营商/企业/消费者三大BG独立运营 |
| 平台化（2012+） | 前台+中台+后台 | 前端灵活作战，中台能力支撑，后台管控风险 |

### 当前组织架构特征

```
           ┌─── 运营商BG ───┐
           ├─── 企业BG ─────┤  前台（作战单元）
           ├─── 消费者BG ───┤
           └─── 云BU ───────┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
  2012实验室   供应链平台    全球服务    中台（能力平台）
    │             │             │
    └─────────────┼─────────────┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
   财经平台     HR平台      流程IT     后台（管控平台）
    └─────────────┘
```

## 岗位体系设计

### 岗位族的划分

| 岗位族 | 典型岗位 | 核心能力 | 任职通道 |
|---|---|---|---|
| 研发技术 | 架构师/工程师/测试 | 技术创新、问题解决 | 技术通道T1-T6 |
| 产品管理 | 产品经理/产品总监 | 市场洞察、产品规划 | 管理通道M1-M5 |
| 市场销售 | 客户经理/解决方案 | 客户关系、商务谈判 | 管理通道M1-M5 |
| 供应链 | 采购/计划/物流 | 供应商管理、计划协同 | 技术通道T1-T5 |
| 专业职能 | HR/财务/法务 | 专业能力、业务支撑 | 技术通道T1-T5 |

### 岗位族划分的原则

1. **工作性质相近**：同类工作归入同一族
2. **能力要求相似**：所需核心能力一致
3. **发展路径互通**：族内岗位可以横向流动
4. **数量适中**：通常5-8个岗位族，不宜过多

<!-- chunk: 组织设计与岗位体系.岗位说明书-jd-的核心要素 -->
## 岗位说明书（JD）的核心要素

### 标准JD模板

| 要素 | 内容 | 作用 |
|---|---|---|
| 岗位名称 | 标准化命名 | 统一称呼，避免混乱 |
| 所属岗位族 | 岗位族+子类 | 归类管理 |
| 汇报关系 | 直属上级/下属 | 明确组织位置 |
| 岗位目的 | 一句话说明存在价值 | 说清为什么设这个岗 |
| 核心职责 | 5-8条关键职责 | 明确做什么 |
| 任职要求 | 知识/技能/经验/行为 | 与任职资格对接 |
| 绩效指标 | 3-5个KPI | 衡量做得好不好 |
| 权限范围 | 决策/审批/资源调配权 | 明确能做到什么 |

### 职责描述的原则

- **用动词开头**：「负责……」「推动……」「确保……」
- **有结果导向**：不只是做什么，还要达到什么效果
- **不重叠不遗漏**：同一层级岗位之间职责边界清晰
- **可衡量**：职责对应的产出可被衡量

## 编制管理

### 编制的定义与计算

**编制 = 组织在预算期内需要的岗位数量**

华为的编制管理逻辑：

```
业务目标 → 工作量估算 → 岗位需求 → 编制确定
    ↑                                        ↓
    └──── 绩效反馈（效率提升→编制压缩）←────┘
```

### 编制管理原则

| 原则 | 说明 |
|---|---|
| 以产出定编制 | 不看有多少人，看需要多少产出 |
| 增量不增人 | 效率提升优先于增加编制 |
| 弹性编制 | 忙季可增加临时编制，淡季回归 |
| 冻结释放 | 离职6个月内可补，超期编制冻结 |

## 组织诊断：常见问题与对策

| 症状 | 根因 | 对策 |
|---|---|---|
| 部门墙厚，协作困难 | 组织按职能设置，缺少横向协调机制 | 建立跨部门团队/项目制 |
| 决策慢，层层汇报 | 层级过多，授权不足 | 扁平化+前端授权 |
| 忙闲不均 | 编制分配不合理 | 工作量分析+编制重新配置 |
| 关键岗位缺人 | 岗位设计不合理/人才梯队断层 | 岗位再设计+继任者计划 |
| 人浮于事 | 职责重叠/流程冗余 | 职责梳理+流程优化 |

## 薄云咨询应用场景

**场景1：组织架构优化**
- 客户特征：公司战略调整，现有组织不匹配
- 方案：战略解读→组织诊断→方案设计→实施推进
- 周期：2-3个月

**场景2：岗位体系建立**
- 客户特征：岗位设置随意，职责不清，人浮于事
- 方案：岗位族划分+岗位分析+JD编写+编制核定
- 周期：2-3个月

**场景3：组织效能提升**
- 客户特征：人效低下，人越多效率越低
- 方案：组织诊断+编制优化+流程简化+授权设计
- 周期：3-4个月
